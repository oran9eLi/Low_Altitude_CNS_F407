# 最小闭环测试方案

版本：V0.2  
日期：2026-06-03  
适用工程：`low_altitude_cns_f407`  

## 1. 目标

本方案用于验证当前框架在未接入真实传感器、未接入正式 LoRa 链路前，是否已经具备一条可观测、可重复的最小运行闭环。

闭环目标如下：

```text
上电启动
  -> USART1 调试串口输出
  -> BSP 初始化
  -> App POST 初始化和自检
  -> FreeRTOS 任务启动
  -> Mock GNSS 传感器输出采样
  -> protocol 任务触发最小闭环演示
  -> 组装 CNS GNSS 调试帧
  -> 串口打印采样值和 CNS 帧 hex dump
```

本测试只验证框架链路，不代表正式通信协议、正式传感器驱动或正式数据上报已经完成。

## 2. 当前边界

本测试覆盖：

| 范围 | 当前实现 |
| --- | --- |
| 调试输出 | USART1，115200-8-N-1 |
| 调试文本编码 | 源码保持 UTF-8，中文调试文本使用 GBK 字节转义 |
| 调试入口 | `Debug/Inc/debug_config.h` 的 `DEBUG_ENABLE` |
| 模拟传感器 | `Sensor_Driver/Src/mock_sensor.c` |
| 采集入口 | `Sensor_RegistryReadAll()` |
| CNS 调试帧组装 | `Debug/Src/debug_cns_demo.c` |
| 通用帧封装 | `App_ProtocolBuildFrame()` |
| 输出方式 | 串口打印 GNSS 采样和 CNS frame hex dump |

本测试不覆盖：

| 不覆盖项 | 说明 |
| --- | --- |
| 真实 GNSS/IMU/Remote ID 驱动 | 当前使用 Mock GNSS |
| LoRa 收发 | E22/LoRa 链路尚未接入此闭环 |
| 正式 `CNS_State` 汇总 | 当前只验证采样到帧的最小路径 |
| 正式 CNS 帧结构 | 当前为调试帧，后续应按 `Docs/protocol` 中正式协议收敛 |
| HMI 显示 | 只保留 display 任务入口和状态打点 |
| SD/CSV 落盘 | 当前只通过串口观察 |

## 3. 文件分工

### 3.1 Debug 目录

```text
Debug/Inc/debug_config.h
Debug/Inc/debug_console.h
Debug/Inc/debug_trace.h
Debug/Inc/debug_cns_demo.h

Debug/Src/debug_console.c
Debug/Src/debug_cns_demo.c
```

| 文件 | 职责 |
| --- | --- |
| `debug_config.h` | 调试总开关，当前启用 `DEBUG_ENABLE` |
| `debug_console.h/.c` | USART1 调试输出、`printf` 重定向、HexDump |
| `debug_trace.h` | 集中管理调试文本和常用打点宏 |
| `debug_cns_demo.h/.c` | 最小闭环演示：读取 Mock 传感器、构造调试 CNS 帧、打印 |

`Debug` 目录是调试基础设施目录，不放入 `Core`，避免污染启动和 HAL 基础层。

### 3.2 App 目录

| 文件 | 职责 |
| --- | --- |
| `App/Src/app_tasks.c` | 创建 FreeRTOS 任务，输出 POST 和任务启动状态 |
| `App/Src/app_protocol.c` | 提供协议解析和通用组帧函数；protocol 任务周期性调用 `DebugCnsDemo_RunOnce()` |
| `App/Inc/app_protocol.h` | 声明 `App_ProtocolBuildFrame()`、`App_ProtocolParse()`、`App_ProtocolTask()` |

### 3.3 Sensor_Driver 目录

| 文件 | 职责 |
| --- | --- |
| `Sensor_Driver/Src/mock_sensor.c` | Mock GNSS 驱动，输出经纬高采样 |
| `Sensor_Driver/Src/sensor_registry.c` | `DEBUG_ENABLE` 下注册 Mock 驱动 |

### 3.4 BSP/Core 相关文件

| 文件 | 职责 |
| --- | --- |
| `Bsp/Src/bsp_uart.c` | 初始化 USART1，提供 `BSP_UART_Send()` |
| `Core/Src/stm32f4xx_hal_msp.c` | 配置 USART1 PA9/PA10、时钟和复用功能 |
| `Core/Src/main.c` | 启动调试串口，输出启动阶段状态 |

## 4. 执行流程

### 4.1 启动阶段

```text
main()
  -> HAL_Init()
  -> SystemClock_Config()
  -> DebugConsole_Init()
  -> BSP_Init()
  -> App_TasksInit()
  -> vTaskStartScheduler()
```

`DebugConsole_Init()` 会先初始化 USART1，并输出调试 banner、串口初始化结果和系统时钟。

### 4.2 任务创建阶段

`App_TasksInit()` 依次完成：

```text
App_StatusInit()
App_AlarmInit()
App_RegistryInitAll()
App_RegistrySelfCheckAll()
xTaskCreate(system)
xTaskCreate(comm)
xTaskCreate(protocol)
xTaskCreate(logger)
xTaskCreate(display)
xTaskCreate(alarm)
```

POST 和任务启动状态通过 `DBG_TRACE_*` 宏输出。

### 4.3 最小闭环演示阶段

`protocol` 任务每 1000 ms 执行一次：

```text
App_ProtocolTask()
  -> DebugCnsDemo_RunOnce()
      -> Sensor_RegistryReadAll()
      -> 判断是否为 SENSOR_VALUE_GEO
      -> 打印 GNSS 采样
      -> DebugCnsDemo_BuildGnssPayload()
      -> App_ProtocolBuildFrame()
      -> DebugConsole_HexDump()
  -> App_LogWrite()
  -> App_StatusHeartbeat(APP_MODULE_PROTOCOL)
```

当前 `DebugCnsDemo_RunOnce()` 是最小闭环测试逻辑，不应扩散到正式协议或传感器模块中。

## 5. 调试帧格式

### 5.1 通用帧格式

当前调试闭环使用 `App_ProtocolBuildFrame()` 生成简化帧：

| 字节 | 字段 | 说明 |
| --- | --- | --- |
| 0 | `0xAA` | 帧头 0 |
| 1 | `0x55` | 帧头 1 |
| 2 | `type` | 帧类型 |
| 3 | `payload_len` | payload 长度 |
| 4..N | `payload` | 负载 |
| 最后 1 字节 | checksum | 从 `type` 到 payload 末尾逐字节 XOR |

### 5.2 GNSS 调试帧

当前 GNSS 调试帧类型：

```c
#define DEBUG_CNS_FRAME_TYPE_GNSS 0x02U
#define DEBUG_CNS_GNSS_PAYLOAD_LEN 25U
```

GNSS payload：

| 偏移 | 长度 | 字段 | 说明 |
| --- | --- | --- | --- |
| 0 | 1 | fix_status | 当前非 0 纬度时填 `2` |
| 1 | 4 | latitude_deg_e7 | 小端，纬度 * 1e7 |
| 5 | 4 | longitude_deg_e7 | 小端，经度 * 1e7 |
| 9 | 4 | altitude_mm | 小端，高度 mm |
| 13 | 4 | timestamp_ms | 小端，FreeRTOS tick |
| 17 | 8 | reserved | 当前填 0 |

该帧只用于最小闭环验证。正式 CNS 通信帧以后应按 `Docs/通信数据结构设计说明.md` 和 `Docs/protocol` 重新实现。

## 10. 常见问题

### 10.1 串口没有任何输出

排查顺序：

1. 确认开发板已上电。
2. 确认串口工具打开的是正确 COM 口。
3. 确认波特率为 115200。
4. 确认使用的是 USART1 对应链路。
5. 临时打开 `DEBUG_BOOT_UART_ONLY`，只验证裸 UART 和 `printf` 循环输出。

### 10.2 有启动输出，但没有任务输出

重点看：

1. `App_TasksInit()` 是否返回 `pdPASS`。
2. 是否进入 `vTaskStartScheduler()`。
3. FreeRTOS heap 是否不足。
4. 是否触发 stack overflow hook。

### 10.3 有任务输出，但没有 GNSS 和 CNS 帧

重点看：

1. `Debug/Inc/debug_config.h` 是否启用 `DEBUG_ENABLE`。
2. `sensor_registry.c` 中 Mock 驱动是否在 `DEBUG_ENABLE` 下注册。
3. `mock_sensor.c` 是否加入 Keil 工程。
4. `DebugCnsDemo_RunOnce()` 是否被 `App_ProtocolTask()` 周期调用。

### 10.4 中文乱码

当前策略：

1. 源文件保持 UTF-8。
2. 中文调试文本在 `debug_trace.h` 中使用 GBK 字节转义。
3. 验收工具优先使用串口调试助手。

如果使用 VS Code 串行监视器，可能因 UTF-8 解码 GBK 字节而乱码，不作为本阶段验收问题。

## 11. 后续收敛方向

当前最小闭环只用于证明框架可运行。正式版本应逐步收敛：

1. 将 Mock GNSS 替换为真实传感器驱动。
2. 将 `DebugCnsDemo_RunOnce()` 中的调试 payload 迁移为正式 App 数据聚合。
3. 由应用层统一汇总 `CNS_State`。
4. 按正式 `CNS_Frame_t` 编码输出。
5. 由通信任务通过 E22 LoRa 链路发送。
6. 调试输出降级为可开关的诊断日志，不参与正式业务路径。

## 12. 当前结论

当前最小闭环已经具备以下能力：

- USART1 调试输出可用。
- FreeRTOS 任务能启动。
- Mock GNSS 能通过传感器注册表输出采样。
- protocol 任务能周期触发闭环演示。
- CNS GNSS 调试帧能组装并通过串口 HexDump 观察。

因此，该闭环可作为后续传感器接入、协议收敛和 LoRa 链路接入前的基础验证用例。
