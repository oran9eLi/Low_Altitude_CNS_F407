# 应用层任务架构调整方案

## 1. 文档范围

本方案只处理主控箱本地应用层闭环，暂不实现以下内容：

- LoRa/E22 收发状态机。
- 主动上报和被动轮询模式。
- Comm 数据出口和空口调度。
- 下行命令、Command 任务和 ACK 重试。
- `g_frame_queue`、`g_cmd_raw_queue`、`g_cmd_queue` 等通信队列。

LoRa 双模式通信另见 `Docs/调整方案/LoRa双模式通信方案.md`，会议评审完成前不纳入本轮代码改动。

本轮目标是先建立稳定的本地数据流：

```text
传感器采集
  ├→ 最新状态快照 → Display / Protocol
  ├→ 每轮完整记录 → Logger / SD
  └→ 异常状态变化 → Alarm
```

## 2. 当前问题

### 2.1 运行期仍周期调用 self_check

当前 `App_RegistryPoll()` 每秒对 SENSOR 注册项执行一次 `App_RegistrySensorSelfCheck()`，最终调用各驱动的 `self_check()` 访问硬件。

这与已经确定的传感器驱动职责冲突：

- `self_check()` 只用于初始化后的启动自检。
- 运行期硬件访问统一由 `read()` 完成。
- `get_status()` 只读取驱动缓存，不主动访问总线。

### 2.2 生产代码没有采集调度入口

`Sensor_RegistryReadAll()` 当前只在调试闭环和单元测试中使用。关闭 `DEBUG_ENABLE` 后，生产代码没有任务周期调用该接口。

### 2.3 采集、显示、日志和协议缺少统一数据源

当前没有应用层采样快照。后续如果 Display、Logger、Protocol 各自读取传感器，会造成：

- 重复访问硬件和总线竞争。
- 各消费者看到的采样时刻不一致。
- 驱动层被显示、日志或协议逻辑反向依赖。

### 2.4 日志无法保存完整采集过程

现有 `App_LogRecord_t` 适合模块运行日志，但不能完整表达一轮采集中的：

- 多条传感器样本。
- 部分成功、部分失败。
- 整轮采集失败且没有样本。
- 具体传感器错误码和 `driver_error`。
- 采集耗时、轮次序号和记录缺口。

日志要求是保留每一轮采集及失败信息，因此 Logger 不能只周期读取最新快照。

### 2.5 传感器告警逻辑仍绑定启动自检

当前传感器 `RAISE/CLEAR` 逻辑位于 `App_RegistrySensorSelfCheck()` 内部。关闭运行期周期自检后，需要独立的运行期健康同步入口。

### 2.6 任务入口分散

System、Comm、Display 任务实现在 `app_tasks.c`，Protocol、Logger、Alarm 在各自模块文件。`app_tasks.c` 同时承担任务编排和具体业务循环，后续维护成本较高。

## 3. 设计原则

### 3.1 状态使用快照，事件使用队列

| 数据 | 语义 | 传递方式 |
|---|---|---|
| 最新采样和系统状态 | 当前事实，可被新数据覆盖 | 互斥保护的 `CNS_State` 快照 |
| 每轮采集记录 | 历史事实，不能被后续采样覆盖 | 固定记录池 + 索引队列 |
| 告警触发和恢复 | 状态变化事件 | 现有 Alarm 消息队列 |
| 普通模块运行日志 | 运行事件 | 现有 Logger 队列 |

快照和队列不互相替代。Display、Protocol 关心最新状态；Logger 需要保存每一轮历史；Alarm 需要保存异常状态变化。

### 3.2 单一硬件访问入口

运行期只有 Acquisition 任务调用 `Sensor_RegistryReadAll()`。Display、Logger、Protocol、Alarm 不直接调用具体传感器驱动，也不主动访问传感器总线。

### 3.3 采集不依赖非关键消费者启动

全部对象必须在 `vTaskStartScheduler()` 前创建。任务启动日志的输出顺序只反映调度顺序，不表示资源创建顺序。

Acquisition 不能无限等待 Display、Logger 或 Protocol，否则非关键模块故障会阻止采集。消费者晚启动时：

- Display 和 Protocol 可以读取最新快照。
- Logger 由采集记录池吸收短时积压。
- Alarm 队列保存异常变化事件。

### 3.4 内存有明确上限

不允许为每轮采集动态分配内存。采集快照、日志记录池和队列容量均采用编译期固定上限，并在落代码后核算 RAM 占用。

## 4. 目标数据流

```text
                     ┌─────────────────┐
                     │ Sensor Registry │
                     │ ReadAll()       │
                     └────────┬────────┘
                              │ 每轮采集
                     ┌────────▼──────────┐
                     │ Acquisition Task  │
                     └───┬────────┬─────┘
                         │        │
            正常/异常样本│        │Sensor_Status_t[]
                         │        │
              ┌──────────▼──┐  ┌──▼────────────────┐
              │ App State   │  │ Sensor Health     │
              │ 最新快照     │  │ RAISE / CLEAR     │
              └────┬────┬───┘  └────────┬──────────┘
                   │    │               │
             ┌─────▼┐ ┌─▼────────┐  ┌──▼────────┐
             │Display│ │ Protocol │  │Alarm Task │
             └──────┘ └──────────┘  └───────────┘

Acquisition Task
       │ 每轮完整 App_AcquisitionRecord_t
       ▼
固定记录池 → 索引队列 → Logger Task → CSV / Storage
```

## 5. Acquisition 模块

### 5.1 文件和职责

新增：

```text
App/Inc/app_acquisition.h
App/Src/app_acquisition.c
```

`App_AcquisitionTask` 负责：

1. 按固定周期调用 `Sensor_RegistryReadAll()`。
2. 为每轮采集生成递增序号、开始时间和耗时。
3. 根据样本和状态确定本轮结果。
4. 更新应用层最新快照。
5. 将完整采集记录提交给 Logger。
6. 将传感器异常状态交给健康同步模块处理。
7. 更新 Acquisition 模块状态和心跳。

初始采集周期使用 `200ms`。多传感器独立采样率调度不纳入本轮。

### 5.2 采集结果

```c
typedef enum {
  APP_ACQ_RESULT_NORMAL = 0,
  APP_ACQ_RESULT_PARTIAL,
  APP_ACQ_RESULT_FAILED
} App_AcquisitionResult_t;
```

判定规则：

| 条件 | 结果 |
|---|---|
| 有样本，所有样本正常，无异常状态 | `NORMAL` |
| 有样本，但存在异常样本或异常状态 | `PARTIAL` |
| 没有任何有效样本 | `FAILED` |

Registry 返回的重要或严重异常不直接覆盖“是否存在有效样本”的事实：存在有效样本时记录为 `PARTIAL`，没有有效样本时记录为 `FAILED`。如果驱动返回异常但没有输出 `Sensor_Status_t`，Acquisition 使用 `ERR_SENSOR_DATA_INVALID` 作为兜底错误，并记录 Registry 返回的严重程度。

### 5.3 完整采集记录

```c
#define APP_ACQ_SAMPLE_MAX 16U
#define APP_ACQ_STATUS_MAX 4U

typedef struct {
  uint32_t sequence;
  uint32_t timestamp_ms;
  uint32_t duration_ms;
  App_AcquisitionResult_t result;
  Sensor_Severity_t severity;
  Sensor_Sample_t samples[APP_ACQ_SAMPLE_MAX];
  uint16_t sample_count;
  uint16_t sample_total;
  Sensor_Status_t status_list[APP_ACQ_STATUS_MAX];
  uint16_t status_count;
  uint16_t status_total;
  uint8_t flags;
} App_AcquisitionRecord_t;
```

每一轮都必须产生一条记录：

- 正常轮次记录全部样本。
- 部分成功记录有效样本及异常状态。
- 完全失败时 `sample_count` 可以为 `0`，但必须保留结果、严重程度和状态信息。
- `sequence` 每轮递增，用于判断日志是否存在缺口。
- `sample_count/status_count` 表示实际存入数组的数量。
- `sample_total/status_total` 表示 Registry 本轮实际返回的总数量。
- `flags` 标记样本截断、状态截断或兜底错误等情况。

如果实际样本数或异常数超过固定容量，记录中保存可容纳部分并设置对应截断标志；不得静默截断。

## 6. 应用层 CNS_State 快照

### 6.1 文件和职责

新增：

```text
App/Inc/app_state.h
App/Src/app_state.c
```

`CNS_State` 是应用层汇总快照，驱动和 BSP 不得直接写入。快照至少包含：

- 最新有效传感器样本及数量。
- 最近一轮采集结果、严重程度和时间戳。
- 各应用模块状态。
- 当前告警摘要。
- 快照版本号或更新时间。

### 6.2 接口边界

```c
void App_StateInit(void);
void App_StateUpdateAcquisition(const App_AcquisitionRecord_t *record);
void App_StateUpdateModuleStatus(App_ModuleId_t module, const App_ModuleStatus_t *status);
void App_StateUpdateAlarmSummary(const App_AlarmRecord_t *record);
BaseType_t App_StateGetSnapshot(CNS_State_t *snapshot);
```

对外只提供更新和复制接口，不暴露可写全局变量。

### 6.3 并发保护

V1.0 采用互斥锁保护快照：

- Acquisition 在完成一轮采集后一次性更新相关字段。
- Display、Protocol、Logger 辅助信息读取时复制完整快照。
- 持锁期间只执行内存复制，不进行 SD、串口、显示或协议编码操作。
- 消费者在释放锁后处理本地副本。

如果后续测得复制耗时影响实时性，再考虑双缓冲；本轮不提前增加复杂度。

## 7. 采集日志链路

### 7.1 日志语义

Logger 必须保留每一轮采集以及失败信息，不能通过周期读取快照代替。

现有 `App_LogRecord_t` 和 `App_LogWrite()` 继续用于普通模块运行日志。新增独立的采集记录提交接口，避免把多样本结构压缩成单个字符串字段。

```c
BaseType_t App_LoggerSubmitAcquisition(const App_AcquisitionRecord_t *record);
```

该接口将记录一次性复制到 Logger 的静态记录池槽位，随后 FreeRTOS 队列只传递槽位索引，不再通过队列复制完整大结构。调用返回后，Acquisition 可以立即复用自己的栈上记录。

### 7.2 固定记录池和索引队列

```text
空闲槽位队列
   ↓ Acquisition 取得槽位索引
固定 App_AcquisitionRecord_t 记录池
   ↓ 填写完成后提交槽位索引
待写入索引队列
   ↓ Logger 取得索引并写 CSV
写入完成后归还空闲槽位队列
```

初始建议：

```c
#define APP_ACQ_RECORD_POOL_SIZE 8U
```

记录池使用静态内存，FreeRTOS 队列元素只保存槽位索引。实施时必须根据 `sizeof(App_AcquisitionRecord_t)` 输出实际 RAM 占用，并结合 24KB FreeRTOS heap 和静态区总量复核。

### 7.3 CSV 表达

一轮采集可以展开成多行 CSV，所有行使用同一 `sequence` 关联：

| `record_type` | 用途 |
|---|---|
| `ROUND` | 每轮固定写一行，保存时间、耗时、结果、严重程度、样本数和异常数 |
| `SAMPLE` | 每个采样点一行，保存 device_id、channel、value_type、value、unit 和 status |
| `STATUS` | 每个异常状态一行，保存 device_id、severity、error_code 和 driver_error |
| `GAP` | 记录由于缓冲耗尽等原因无法保存的轮次区间 |

完全失败的轮次即使没有 `SAMPLE` 行，也必须有一条 `ROUND`，并尽可能写入对应 `STATUS` 行。

### 7.4 缓冲耗尽和存储失败

有限 RAM 无法在 SD 长期不可用时无限保存采样。处理规则如下：

1. 正常运行和 Logger 短时阻塞由固定记录池吸收。
2. 记录池耗尽时，Acquisition 不无限阻塞采集任务。
3. 记录缺失时保存首个缺失序号、最后缺失序号和缺失轮数。
4. Logger 恢复写入后优先补写一条 `GAP` 记录。
5. 触发日志缓冲区满或存储写失败告警，不允许静默丢失。
6. SD 写入失败时记录槽位不能永久占用；记录失败信息后释放槽位，避免整个采集链路最终停滞。

需要在错误码表中补充明确的日志缓冲区满错误，例如 `ERR_LOG_QUEUE_FULL`，并区分 `ERR_LOG_WRITE_FAILED`。

## 8. 传感器健康同步

### 8.1 文件和职责

新增：

```text
App/Inc/app_sensor_health.h
App/Src/app_sensor_health.c
```

公开入口：

```c
void App_SensorHealthSync(const Sensor_Status_t *status_list, uint16_t status_count);
```

职责：

- 将本轮非正常状态转换为统一 `App_AlarmMsg_t` 并投递 `RAISE`。
- 使用 `error_code + sensor_id` 作为传感器活动异常 key。
- 查询当前活动传感器异常，找出本轮已不存在的 key 并投递 `CLEAR`。
- 传感器名称由 Registry 按 `device_id` 查询，不要求驱动重复填充名称。

原 `App_RegistrySensorSelfCheck()` 中的运行期告警同步逻辑迁移到该模块。启动自检仍可以复用相同健康同步入口上报启动异常，但 Registry 不再承担运行期传感器健康维护。

## 9. Display、Protocol 和 Logger 的消费方式

### 9.1 Display

- 周期读取 `CNS_State` 快照。
- 不读取传感器硬件。
- 不解析 CNS 二进制通信帧。
- 当前只保留数据输入和刷新函数签名，由显示负责人接入具体页面变量。

### 9.2 Protocol

- 从 `CNS_State` 获取最新状态和最新采样。
- 负责 CNS STATUS、SENSOR、ALARM 等消息编码。
- 不调用 `Sensor_RegistryReadAll()`。
- 当前没有 Comm 出口时，仅保留编码接口和调试验证入口。
- 不建立 LoRa 发送队列和下行命令队列。

### 9.3 Logger

- 通过采集记录池逐轮写入采集 CSV。
- 继续通过现有普通日志队列处理系统运行日志。
- 不依赖快照保存采集历史。
- 可以读取快照补充模块状态，但快照不能替代采集记录。

## 10. Ready 通知

保留 FreeRTOS EventGroup 作为任务就绪状态通知，但不建立全任务同步屏障。

```c
#define APP_READY_BIT_ACQUISITION (1U << 0)
#define APP_READY_BIT_PROTOCOL    (1U << 1)
#define APP_READY_BIT_LOGGER      (1U << 2)
#define APP_READY_BIT_DISPLAY     (1U << 3)
#define APP_READY_BIT_ALARM       (1U << 4)
```

规则：

- 任务完成自身任务上下文初始化后设置 Ready 位。
- System 可以读取 Ready 位进行启动状态汇总和调试输出。
- Acquisition 在自身初始化完成后即可开始周期采集，不等待所有消费者。
- Logger 的记录池和队列必须在任务创建前完成初始化，保证 Logger 任务尚未获得调度时也能暂存记录。
- Ready 位不表达队列是否创建，所有跨任务对象必须在 `App_TasksInit()` 创建任务前完成创建。

## 11. 任务组织

### 11.1 目标文件归属

| 任务 | 定义位置 | 职责 |
|---|---|---|
| `App_SystemTask` | `app_system.c` | 注册表 Poll、任务状态汇总、系统心跳 |
| `App_AcquisitionTask` | `app_acquisition.c` | 周期采集、快照更新、日志提交、健康同步 |
| `App_ProtocolTask` | `app_protocol.c` | 从快照编码 CNS 数据，保留调试验证 |
| `App_LoggerTask` | `app_logger.c` | 普通日志和逐轮采集日志写入 |
| `App_DisplayTask` | `app_display.c` | 从快照刷新 HMI 入口 |
| `App_AlarmTask` | `app_alarm.c` | 处理异常状态变化 |
| `App_CommTask` | 暂时保留现状 | 本轮不接入 LoRa 链路设计 |

### 11.2 app_tasks.c 职责

`app_tasks.c` 收缩为应用层编排入口：

```text
App_TasksInit()
  1. 初始化 App_Status、App_State 和 Alarm
  2. 初始化 Logger 普通队列、采集记录池和索引队列
  3. 创建 Ready EventGroup
  4. 执行 App_RegistryInitAll()
  5. 执行 App_RegistrySelfCheckAll()
  6. 创建各应用任务
  7. 任一步失败则返回 pdFAIL
```

## 12. 注册表调整

SENSOR 注册项调整为：

```c
{
  APP_MODULE_SENSOR,
  "sensor",
  App_RegistrySensorInit,
  App_RegistrySensorSelfCheck,
  NULL,
  0U
}
```

含义：

- 初始化阶段调用 `init()`。
- 启动自检阶段调用一次 `self_check()`。
- `App_RegistryPoll()` 不再周期调用传感器自检。
- 运行期状态由 Acquisition 和 Sensor Health 维护。

新增 `APP_MODULE_ACQUISITION`，用于记录采集任务心跳和整体采集状态。是否把它加入静态模块注册表取决于是否需要独立初始化入口；如果没有初始化和自检回调，可以只加入 `App_ModuleId_t` 和状态表，不增加空注册项。

## 13. 队列和内存策略

本轮只新增采集日志所需的两个索引队列：

| 对象 | 元素 | 用途 |
|---|---|---|
| 空闲槽位队列 | 记录池索引 | Acquisition 获取可写槽位 |
| 待写入队列 | 记录池索引 | Logger 获取待写记录 |

以下对象本轮不新增：

- 完整 `App_AcquisitionRecord_t` 数据队列。
- Display 数据队列。
- Protocol 帧发送队列。
- Comm 接收队列。
- Command 队列。

Display 和 Protocol 通过快照解耦；Alarm 继续使用现有消息队列；普通运行日志继续使用现有 Logger 队列。

## 16. 验证要求

### 16.1 单元和主机侧测试

- 正常采集产生 `NORMAL` 记录。
- 部分样本异常产生 `PARTIAL` 记录。
- 无样本且存在错误产生 `FAILED` 记录。
- 每轮 `sequence` 连续递增。
- 快照只保存最新一轮数据。
- Logger 按轮次写出 ROUND/SAMPLE/STATUS。
- 记录池耗尽时产生 GAP 和告警，不静默丢失。
- 传感器恢复时产生对应 CLEAR。
- Display、Protocol 不直接调用 Sensor Registry。

### 16.2 板级验证

- 启动自检后不再每秒调用传感器 `self_check()`。
- 200ms 采集任务持续运行。
- Display/调试输出能够看到最新采样。
- 每一轮成功、部分失败和完全失败都能在 CSV 中定位。
- 临时拔掉传感器后记录失败，恢复后告警清除且采集继续。
- 模拟 SD 短时阻塞，记录池能够吸收积压。
- Keil 构建结果达到 `0 Error(s), 0 Warning(s)`。

## 17. 结论

本轮应用层调整不处理 LoRa 链路，而是先完成主控箱本地数据闭环：

```text
Acquisition 统一采集
  → CNS_State 保存最新状态
  → Logger 保存每轮历史和失败信息
  → Alarm 保存异常状态变化
  → Display / Protocol 读取同一份最新快照
```

该结构同时满足实时展示、后续协议上报和实验过程追溯，并为 LoRa 通信方案评审完成后的 Comm 接入保留稳定的数据来源。
