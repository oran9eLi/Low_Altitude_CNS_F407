# 统一异常处理链路重构方案

## 目标

将当前“告警项 + 标准错误码 + 多套触发 API + 位图记录”的告警机制，收敛为“统一异常状态变化消息 + 模块化错误码 + payload 联合体 + 队列消费”的异常处理链路。

改造后的目标是：

- 各模块按统一消息格式上报异常状态变化。
- 普通异常统一进入异常队列，由异常处理任务统一消费。
- 错误码按模块分段管理，面向 HMI、日志、CNS_State 和 LoRa 上报。
- payload 只携带现场上下文，不再承担“标准错误码”的职责。
- 传感器错误码抽象为关键业务错误，不暴露 I2C、SPI、UART 等底层总线细节。
- 传感器 payload 带上传感器 `id` 和 `name`，便于上层显示、日志和上报。

## 当前问题

### API 入口过多

当前告警模块存在多套入口：

```c
App_AlarmBuildRaiseMsg()
App_AlarmBuildRaiseDefaultMsg()
App_AlarmPost()
App_AlarmRaise()
App_AlarmRaiseDefault()
App_AlarmRaiseImmediate()
```

调用方需要判断“直接 Raise 还是先 Build 再 Post”，接口选择成本偏高。后续如果每个模块都要携带不同 payload，这种双路径会继续扩大复杂度。

### 告警项和错误码职责重叠

当前 `App_AlarmId_t` 表达内部告警项，`App_ErrorCode_t` 表达对外错误码，但很多场景下二者是一一映射关系，调用方很容易不清楚该传告警项还是错误码。

例如：

```text
APP_ALARM_GNSS_NO_FIX
APP_ERROR_GNSS_NO_FIX
```

这会形成两套需要维护的一致性表。

### 位图不适合实例化异常

`s_alarm_mask` 依赖 `App_AlarmId_t` 作为位号。它适合少量固定告警项，但不适合区分多个传感器实例、多个通信链路或多个存储文件异常。

后续如果两个传感器都触发同一种错误，仅靠 `APP_ALARM_SENSOR_FAULT` 位图无法表达是哪一个设备。

### 传感器错误暴露过细

当前存在 `APP_ERROR_SENSOR_I2C` 这类错误码。对上层而言，传感器异常更应该表达业务可理解的关键错误，例如离线、无数据、数据无效、GNSS 无定位等。I2C、SPI、UART、芯片状态等底层细节应进入 payload 的 `detail` 字段。

## 总体设计

### 统一异常消息

普通模块不再直接操作告警记录，也不再选择多套告警触发 API。目标路径统一为：

```text
模块发现异常状态变化
  -> 构造 App_AlarmMsg_t
  -> App_AlarmPost()
  -> 告警/异常队列
  -> App_AlarmTask()
  -> 活动异常表 + 模块状态
  -> HMI / 日志 / CNS_State / LoRa 上报查询
```

为降低改动面，命名可以继续保留 `App_Alarm` 和 `App_AlarmMsg_t`。这里的“Alarm”不再只表示固定告警项，而表示统一异常/告警事件链路。

```text
App_Alarm      保留
App_AlarmMsg_t 保留，但语义改为统一异常/告警状态变化消息
App_AlarmTask  保留，继续作为异常/告警事件队列消费任务
```

因此，后续方案不强制重命名文件和模块，只重构消息结构、错误码体系、payload 结构和记录表职责。

### 消息头承载操作类型和标准错误码

错误码应放在消息头，不建议放在 payload 中。

原因：

- HMI、日志、CNS_State、LoRa 上报无需解析 payload 就能知道错误类型。
- payload 是联合体，不同模块格式不同，不适合作为统一错误码入口。
- 队列消费、活动异常表去重和当前错误选择都需要稳定、统一的 key。

目标结构：

```c
typedef enum {
  APP_ALARM_OP_RAISE = 0,
  APP_ALARM_OP_CLEAR
} App_AlarmOp_t;

typedef struct
{
  App_AlarmOp_t op;
  App_ModuleId_t source;
  App_AlarmSeverity_t severity;
  App_ErrorCode_t code;
  App_AlarmPayloadType_t payload_type;
  uint16_t instance_id;
  uint32_t timestamp_ms;
} App_AlarmMsgHeader_t;

typedef struct
{
  App_AlarmMsgHeader_t header;
  App_AlarmPayload_t payload;
} App_AlarmMsg_t;
```

其中：

- `op`：异常状态变化类型，`RAISE` 表示进入或持续异常，`CLEAR` 表示异常恢复。
- `source`：异常来源模块。
- `severity`：异常严重程度。
- `code`：标准错误码，面向显示、日志和上报。
- `payload_type`：payload 联合体有效分支。
- `instance_id`：同一模块下的实例编号，例如传感器 ID、链路 ID、文件 ID。
- `timestamp_ms`：异常状态变化时间。

### payload 承载现场上下文

payload 不负责表达“发生了什么标准错误”，只负责表达“这个错误发生时的现场信息”。

例如传感器：

```c
#define APP_ALARM_NAME_LEN 16U

typedef struct
{
  uint16_t sensor_id;
  uint16_t sensor_type;
  char name[APP_ALARM_NAME_LEN];
  uint32_t detail;
} App_AlarmSensorPayload_t;
```

其中：

- `sensor_id`：传感器设备编号。
- `sensor_type`：GNSS、IMU、电源等类型。
- `name`：固定长度名称，便于上层直接展示。
- `detail`：驱动细节，例如 HAL 返回值、芯片状态或底层总线错误。

## 错误码体系

### 类型归属

错误码不属于告警模块，应放在独立公共头文件中。当前可以沿用 `App/Inc/app_error.h`，但该文件只定义错误码类型和 `ERR_` 枚举，不包含告警队列、payload、记录表或模块状态接口。

目标类型名使用：

```c
App_ErrorCode_t
```

不再使用 `App_AlarmCode_t`。传感器驱动层如果需要填充统一错误码，只包含这个轻量错误码头文件，不包含 `app_alarm.h`。

### 编码方式

错误码可以使用十六进制，也可以使用十进制。编码显示形式不是关键，关键是按模块分段、语义稳定、便于上层显示和协议上报。

如果继续沿用 `0x0000` 这类写法，也没有问题。建议保留模块分段思路，例如：

```text
0x0000       OK
0x0100-0x01FF  System
0x0200-0x02FF  Sensor
0x0300-0x03FF  Comm / LoRa
0x0400-0x04FF  Display / HMI
0x0500-0x05FF  Storage / Logger
0x0600-0x06FF  Protocol
0x0F00-0x0FFF  Unknown / Reserved
```

如果使用十进制，也可以采用 `100-199`、`200-299` 这类分段。不要直接套用 HTTP 语义，例如 `404` 容易被理解为 not found；错误码含义必须由项目错误码表定义。

### 目标枚举

```c
typedef enum {
  ERR_OK = 0x0000U,

  ERR_SYSTEM_SELF_CHECK_FAILED = 0x0101U,
  ERR_SYSTEM_HEAP_FAILED       = 0x0102U,
  ERR_SYSTEM_STACK_OVERFLOW    = 0x0103U,
  ERR_SYSTEM_ALARM_QUEUE_FULL  = 0x0104U,
  ERR_SYSTEM_ALARM_RECORD_FULL = 0x0105U,

  ERR_SENSOR_OFFLINE           = 0x0201U,
  ERR_SENSOR_NO_DATA           = 0x0202U,
  ERR_SENSOR_DATA_INVALID      = 0x0203U,
  ERR_SENSOR_NO_FIX            = 0x0204U,
  ERR_SENSOR_INIT_FAILED       = 0x0205U,
  ERR_SENSOR_TIMEOUT           = 0x0206U,

  ERR_COMM_LORA_OFFLINE        = 0x0301U,
  ERR_COMM_TIMEOUT             = 0x0302U,
  ERR_COMM_FRAME_INVALID       = 0x0303U,

  ERR_DISPLAY_OFFLINE          = 0x0401U,
  ERR_DISPLAY_REFRESH_FAILED   = 0x0402U,

  ERR_STORAGE_NOT_READY        = 0x0501U,
  ERR_STORAGE_WRITE_FAILED     = 0x0502U,
  ERR_LOG_WRITE_FAILED         = 0x0503U,

  ERR_PROTOCOL_PARSE_FAILED    = 0x0601U,
  ERR_PROTOCOL_CHECKSUM_FAILED = 0x0602U,

  ERR_UNKNOWN                  = 0x0FFFU
} App_ErrorCode_t;
```

### 传感器错误码边界

传感器错误码只抽关键错误，不按底层总线拆分：

| 错误码 | 含义 | 不包含的细节 |
| --- | --- | --- |
| `ERR_SENSOR_OFFLINE` | 传感器离线或长时间无响应 | I2C NACK、UART 超时、SPI 状态错误 |
| `ERR_SENSOR_NO_DATA` | 当前没有可用数据 | 具体寄存器状态、缓冲区状态 |
| `ERR_SENSOR_DATA_INVALID` | 数据校验失败、格式错误或数值不可信 | CRC 错误、字段范围错误 |
| `ERR_SENSOR_NO_FIX` | GNSS 无有效定位 | 卫星数、定位状态原始值 |
| `ERR_SENSOR_INIT_FAILED` | 初始化失败 | HAL 初始化返回值 |
| `ERR_SENSOR_TIMEOUT` | 采样或通信超时 | 具体总线类型和超时时间 |

传感器驱动层可以直接使用这套统一错误码，但边界必须收住：

- 驱动只允许返回 `ERR_SENSOR_*` 范围内的错误码。
- 驱动不直接构造 `App_AlarmMsg_t`，也不调用 `App_AlarmPost()`。
- 驱动不依赖完整告警模块，只包含轻量错误码头文件。
- 注册表不再维护 `Sensor_FaultReason_t -> App_ErrorCode_t` 映射，只校验并透传 `status.code`。
- 如果 `severity != SENSOR_SEVERITY_NORMAL` 但 `code == ERR_OK`，注册表应兜底改为 `ERR_SENSOR_DATA_INVALID` 或 `ERR_UNKNOWN`。

目标传感器状态结构：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Type_t sensor_type;
  Sensor_Severity_t severity;
  App_ErrorCode_t code;
  uint32_t driver_error;
} Sensor_Status_t;
```

其中 `code` 是全系统统一错误码，必须由驱动按当前状态填写。`driver_error` 是可选调试细节，不是驱动必须提供的标准错误事实。

`driver_error` 的建议规则：

- 正常时填 `0U`。
- 驱动能拿到底层细节时，可以填 HAL 返回值、`HAL_xxx_GetError()`、芯片状态寄存器、协议解析错误或驱动自定义阶段码。
- 驱动暂时没有底层细节时，可以填 `0U`，不影响上层判断。
- 上层业务、HMI、日志和上报的主错误判断只看 `code` 和 `severity`，不能依赖 `driver_error`。

底层细节放入：

```text
payload.sensor.detail
```

## 严重程度

应用层异常严重程度与传感器模块保持一致的语义：

```c
typedef enum {
  APP_ALARM_SEVERITY_NORMAL = 0,
  APP_ALARM_SEVERITY_GENERAL,
  APP_ALARM_SEVERITY_IMPORTANT,
  APP_ALARM_SEVERITY_CRITICAL
} App_AlarmSeverity_t;
```

原 `App_CheckResult_t` 也应同步改为：

```c
typedef enum {
  APP_CHECK_NORMAL = 0,
  APP_CHECK_GENERAL,
  APP_CHECK_IMPORTANT,
  APP_CHECK_CRITICAL
} App_CheckResult_t;
```

建议映射：

```text
APP_CHECK_NORMAL    -> APP_ALARM_SEVERITY_NORMAL
APP_CHECK_GENERAL   -> APP_ALARM_SEVERITY_GENERAL
APP_CHECK_IMPORTANT -> APP_ALARM_SEVERITY_IMPORTANT
APP_CHECK_CRITICAL  -> APP_ALARM_SEVERITY_CRITICAL
```

## 活动异常记录表

原 `s_alarm_mask` 不适合继续作为主状态结构。目标改为固定长度活动异常记录表。

### 记录 key

活动异常的唯一 key：

```text
source + code + instance_id
```

示例：

```text
APP_MODULE_SENSOR + ERR_SENSOR_NO_FIX + GNSS_SENSOR_ID
APP_MODULE_COMM   + ERR_COMM_TIMEOUT  + LORA_LINK_ID
APP_MODULE_SYSTEM + ERR_SYSTEM_HEAP_FAILED + 0
```

`active_records[]` 使用固定容量数组和线性查找。当前记录容量较小，线性扫描更简单、更确定，也更适合调试和嵌入式资源约束。

### 记录结构

```c
typedef struct
{
  uint8_t active;
  App_ModuleId_t source;
  App_AlarmSeverity_t severity;
  App_ErrorCode_t code;
  App_AlarmPayloadType_t payload_type;
  uint16_t instance_id;
  uint32_t first_tick_ms;
  uint32_t last_tick_ms;
  App_AlarmPayload_t payload;
} App_AlarmRecord_t;
```

`count` 不再保留。周期自检型异常会在每轮自检中重复确认，简单累计次数只能反映自检执行次数，不能反映异常真实发生次数，容易误导判断。

字段语义：

- `first_tick_ms`：第一次进入异常状态的时间。
- `last_tick_ms`：最近一次确认异常仍存在的时间。
- `payload`：最近一次异常现场信息。
- `active`：该记录当前是否仍处于异常状态。

### 状态变化规则

告警任务根据 `op + source + code + instance_id` 更新活动异常表：

```text
RAISE + 已存在相同 key:
  刷新 last_tick_ms、severity、payload

RAISE + 不存在相同 key:
  分配空记录，写入 first_tick_ms、last_tick_ms、severity、payload

CLEAR + 已存在相同 key:
  清除该活动记录

CLEAR + 不存在相同 key:
  忽略，不产生新异常
```

`RAISE` 表示异常进入或持续存在，不代表“新增一次异常”。`CLEAR` 表示该 key 已恢复正常。

### 并发保护

队列本身由 FreeRTOS 保证线程安全，但 `active_records[]` 会被告警任务写入，也会被 HMI、日志、CNS_State 汇总和注册表恢复判断读取，因此记录表访问必须加保护。

建议：

- 普通路径下，`App_AlarmTask()` 是活动异常表的唯一写入者。
- 对外查询接口在内部使用互斥锁或临界区复制快照。
- 查询方只读取复制出来的记录，不直接持有 `active_records[]` 指针。
- 系统 hook 属于异常路径，内部静态记录函数进入临界区后直接写表，不经过队列，也不作为普通模块接口暴露。

### 表容量

建议先固定容量：

```c
#define APP_ALARM_ACTIVE_MAX 16U
```

V1.0 阶段异常来源有限，固定表比动态内存更安全。表满时应保留严重程度最高的记录，或保留已有记录并丢弃低等级新异常，同时设置系统级记录表溢出异常。

### 队列和记录表溢出

异常队列满或活动记录表满不能静默丢失。

建议处理：

- `App_AlarmPost()` 投递失败时返回 `pdFAIL`，调用方可设置当前模块状态为 `ERR_SYSTEM_ALARM_QUEUE_FULL`。
- 告警任务发现 `active_records[]` 无可用槽位时，记录 `ERR_SYSTEM_ALARM_RECORD_FULL`。
- 如果记录表已满且无法写入新的系统级溢出异常，应至少保留已有高严重程度异常，并在调试日志中输出失败原因。

## 目标 API

### 主路径

```c
void App_AlarmInit(void);
App_AlarmResult_t App_AlarmBuildMsg(App_AlarmMsg_t *msg,
                                    App_AlarmOp_t op,
                                    App_ModuleId_t source,
                                    App_AlarmSeverity_t severity,
                                    App_ErrorCode_t code,
                                    uint16_t instance_id,
                                    App_AlarmPayloadType_t payload_type,
                                    const App_AlarmPayload_t *payload);
BaseType_t App_AlarmPost(const App_AlarmMsg_t *msg, TickType_t timeout_ticks);
BaseType_t App_AlarmPostFromISR(const App_AlarmMsg_t *msg, BaseType_t *higher_priority_task_woken);
uint32_t App_AlarmProcessPending(uint32_t max_messages);
void App_AlarmTask(void *argument);
```

### 查询接口

```c
uint16_t App_AlarmGetActiveCount(void);
App_AlarmResult_t App_AlarmGetRecord(uint16_t index, App_AlarmRecord_t *record);
uint16_t App_AlarmFindBySourceInstance(App_ModuleId_t source,
                                       uint16_t instance_id,
                                       App_AlarmRecord_t *records,
                                       uint16_t max_count);
App_ErrorCode_t App_AlarmGetCurrentCode(void);
App_AlarmSeverity_t App_AlarmGetCurrentSeverity(void);
```

### 清除消息

清除也通过统一消息队列完成，不直接修改活动异常表。

```text
op          = APP_ALARM_OP_CLEAR
source      = 原异常来源模块
severity    = APP_ALARM_SEVERITY_NORMAL
code        = 原异常错误码
instance_id = 原异常实例编号
payload_type = APP_ALARM_PAYLOAD_NONE
payload     = none
```

告警任务收到 `CLEAR` 后，按 `source + code + instance_id` 查找并清除对应活动记录。

### 删除旧接口和旧结构

以下接口和结构直接删除，避免继续维持两套异常入口：

```c
App_AlarmRaise()
App_AlarmRaiseDefault()
App_AlarmBuildRaiseDefaultMsg()
App_AlarmGetActiveMask()
App_AlarmRaiseImmediate()
App_AlarmId_t
s_alarm_mask
```

删除原则：

- 调用方统一构造 `App_AlarmMsg_t` 并通过 `App_AlarmPost()` 或 `App_AlarmPostFromISR()` 投递。
- 活动异常只由 `active_records[]` 表达，不再维护告警项位图。
- 告警项不再作为独立数据结构存在，错误事实由 `App_ErrorCode_t` 表达。
- 系统 hook 如需绕过队列，应使用告警模块内部的静态记录函数，不对外暴露旧立即记录接口。

## 目标异常链路

### 传感器异常

```text
传感器驱动
  -> Sensor_Status_t
  -> Sensor_RegistrySelfCheckAll()
  -> App_RegistrySensorSelfCheck()
  -> App_AlarmBuildMsg(
       op          = APP_ALARM_OP_RAISE,
       source      = APP_MODULE_SENSOR,
       severity    = Sensor_Severity_t 映射结果,
       code        = status.code,
       instance_id = sensor_id,
       payload     = sensor_id + sensor_type + name + detail
     )
  -> App_AlarmPost()
  -> App_AlarmTask()
  -> active_records[]
  -> App_Status
  -> HMI / 日志 / CNS_State / LoRa 上报
```

同一轮自检中如果多个传感器异常，每个异常传感器各构造并投递一条 `App_AlarmMsg_t`。不合并成汇总消息。

```text
APP_MODULE_SENSOR + ERR_SENSOR_NO_FIX   + GNSS_SENSOR_ID -> 一条 RAISE
APP_MODULE_SENSOR + ERR_SENSOR_TIMEOUT  + IMU_SENSOR_ID  -> 一条 RAISE
APP_MODULE_SENSOR + ERR_SENSOR_NO_DATA  + BARO_SENSOR_ID -> 一条 RAISE
```

传感器状态示例：

| 驱动观察到的事实 | `status.code` | payload |
| --- | --- | --- |
| GNSS 无有效定位 | `ERR_SENSOR_NO_FIX` | `sensor_id`、`name`、`detail` |
| 传感器离线或长时间无响应 | `ERR_SENSOR_OFFLINE` | `sensor_id`、`name`、`detail` |
| 当前没有可用数据 | `ERR_SENSOR_NO_DATA` | `sensor_id`、`name`、`detail` |
| 数据校验失败、格式错误或数值不可信 | `ERR_SENSOR_DATA_INVALID` | `sensor_id`、`name`、`detail` |
| 初始化失败 | `ERR_SENSOR_INIT_FAILED` | `sensor_id`、`name`、`detail` |
| 采样或通信超时 | `ERR_SENSOR_TIMEOUT` | `sensor_id`、`name`、`detail` |

### 传感器恢复

告警模块无法主动判断传感器恢复，恢复必须由注册表根据新一轮自检结果判断后投递 `CLEAR` 消息。

```text
App_RegistrySensorSelfCheck()
  -> 读取本轮异常 status_list[]
  -> 查询告警模块中 APP_MODULE_SENSOR 的活动异常记录
  -> 对比当前异常 key 和活动异常 key
  -> 当前仍异常的 key:
       投递 RAISE，刷新 last_tick_ms 和 payload
  -> 活动表中存在、本轮不存在的 key:
       投递 CLEAR
  -> 同一传感器异常类型变化:
       CLEAR 旧 key
       RAISE 新 key
```

示例：

```text
上一轮: APP_MODULE_SENSOR + ERR_SENSOR_NO_FIX  + GNSS_SENSOR_ID
本轮:   GNSS_SENSOR_ID 正常
结果:   投递 CLEAR(APP_MODULE_SENSOR, ERR_SENSOR_NO_FIX, GNSS_SENSOR_ID)

上一轮: APP_MODULE_SENSOR + ERR_SENSOR_NO_FIX  + GNSS_SENSOR_ID
本轮:   APP_MODULE_SENSOR + ERR_SENSOR_TIMEOUT + GNSS_SENSOR_ID
结果:   先投递 CLEAR 旧 key，再投递 RAISE 新 key
```

恢复判断由 `App_RegistrySensorSelfCheck()` 负责。传感器驱动只输出当前状态，不维护历史异常，也不直接清除告警。

### 显示异常

```text
Display_SelfCheck() / Display_Refresh()
  -> Display_Result_t + App_ErrorCode_t
  -> App_AlarmBuildMsg(
       op          = APP_ALARM_OP_RAISE,
       source      = APP_MODULE_DISPLAY,
       severity    = APP_ALARM_SEVERITY_GENERAL 或 IMPORTANT,
       code        = ERR_DISPLAY_OFFLINE / ERR_DISPLAY_REFRESH_FAILED,
       instance_id = 0,
       payload     = raw_detail 或后续 display payload
     )
  -> App_AlarmPost()
```

显示恢复时也投递 `CLEAR`：

```text
Display_Refresh() 返回 DISPLAY_OK
  -> CLEAR(APP_MODULE_DISPLAY, ERR_DISPLAY_OFFLINE, 0)
  -> CLEAR(APP_MODULE_DISPLAY, ERR_DISPLAY_REFRESH_FAILED, 0)
```

### 存储和日志异常

```text
App_StorageFlush() / App_StorageWriteCsvLine()
  -> App_StorageResult_t
  -> App_AlarmBuildMsg(
       op          = APP_ALARM_OP_RAISE,
       source      = APP_MODULE_STORAGE 或 APP_MODULE_LOGGER,
       severity    = APP_ALARM_SEVERITY_IMPORTANT,
       code        = ERR_STORAGE_NOT_READY / ERR_LOG_WRITE_FAILED,
       instance_id = file_index 或 0,
       payload     = storage operation + detail
     )
  -> App_AlarmPost()
```

存储或日志恢复时，按原异常 key 投递 `CLEAR`。例如：

```text
App_StorageFlush() 恢复 APP_STORAGE_OK
  -> CLEAR(APP_MODULE_STORAGE, ERR_STORAGE_NOT_READY, 0)

日志写入恢复成功
  -> CLEAR(APP_MODULE_LOGGER, ERR_LOG_WRITE_FAILED, 0)
```

### 系统异常

```text
vApplicationMallocFailedHook()
vApplicationStackOverflowHook()
  -> 告警模块内部静态记录函数
  -> active_records[]
  -> App_Status
```

系统 hook 不经过队列，避免调度器或堆状态异常时无法投递。该路径只作为告警模块内部实现，不导出旧立即记录 API。

## 迁移步骤

1. 重构 `App_ErrorCode_t`，将其作为独立公共错误码类型，复用或调整 `App_AlarmSeverity_t`，同步删除旧错误码体系中不再需要的定义。
2. 修改 `App_CheckResult_t`，统一模块自检严重程度。
3. 重构 `App_AlarmMsg_t`、payload 联合体和活动异常记录表，消息头增加 `APP_ALARM_OP_RAISE/CLEAR`。
4. 保留 `App_AlarmPost()` 作为主投递入口，但消息语义改为统一异常/告警状态变化事件。
5. 修改传感器状态结构，将 `Sensor_Status_t.reason` 替换为 `Sensor_Status_t.code`。
6. 改造 `App_Status`，所有模块状态统一保存 `App_ErrorCode_t`。
7. 先改传感器异常链路，因为它最需要 payload 和实例化异常；注册表直接透传传感器错误码，不再做二次映射。
8. 在注册表中实现传感器恢复判断：仍异常投递 RAISE，已恢复投递 CLEAR，异常类型变化时 CLEAR 旧 key 后 RAISE 新 key。
9. 再改显示、存储、日志、系统 hook。
10. 移除 `App_AlarmId_t`、`s_alarm_mask`、`count` 和旧 Raise 系列接口。
11. 更新协议错误码表、CNS_State、HMI 变量和测试方案文档。
12. Keil 构建验证，目标 `0 Error(s), 0 Warning(s)`。

## 方案结论

该方案可行，并且比当前告警模块更适合后续扩展。

关键判断：

- 用统一异常消息替代多套 Raise/Build API，可以明显降低调用方复杂度。
- 用模块化错误码替代“告警项 + 错误码”双体系，可以减少重复映射。
- 用活动异常记录表替代位图，可以支持多个传感器、多个链路和多个实例。
- 传感器错误码抽象为关键业务错误，可以避免上层绑定具体总线实现。
- 统一错误码可以暴露给传感器驱动层，驱动只填 `ERR_SENSOR_*`，注册表透传。
- payload 带 `sensor_id` 和 `name`，能满足显示、日志和上报的直观性。

需要坚持的边界：

- 错误码放在消息头。
- payload 放现场上下文。
- 普通模块只投递异常消息，不直接写异常记录。
- 系统 hook 只允许走告警模块内部静态记录路径，不导出额外旧接口。
