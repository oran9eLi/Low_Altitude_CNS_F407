# 传感器驱动层调整方案

## 1. 目标

当前传感器驱动层已经具备统一句柄、采样结构和异常状态输出，但 `self_check`、`read`、`get_status` 的职责边界仍不够清晰。运行期周期自检可能与采样访问同一硬件和同一总线，带来重复访问、状态来源不一致和总线竞争风险。

本次调整目标：

- 运行期硬件访问入口收敛到采样链路，避免周期自检重复访问总线。
- 驱动内部维护当前状态缓存，应用层通过统一状态快照判断异常触发和恢复。
- `get_status` 只读取缓存，不访问硬件。
- `Sensor_Sample_t.status` 使用布尔语义，只表达本次样本是否异常，不承载具体错误原因。
- 移除运行期结构中的 `sensor_type`，运行期链路只保留必要识别字段。

## 2. 当前问题

### 2.1 自检和采样职责重合

大多数传感器异常会在采样时自然暴露，例如：

- 总线超时或设备无响应。
- 数据为空或采样数量为 0。
- 数据帧校验失败。
- 数据字段格式错误或超出物理合理范围。
- GNSS 无定位。

如果运行期再周期调用 `self_check`，通常也需要读寄存器、发命令或访问串口/I2C/SPI，这与采样动作重叠。

### 2.2 总线访问可能竞争

当前设计允许多个入口访问硬件：

```text
App 周期自检 -> Sensor_RegistrySelfCheckAll() -> driver->self_check()
采样任务      -> Sensor_RegistryReadAll()           -> driver->read()
状态查询      -> driver->get_status()
```

如果 `self_check` 和 `read` 在不同任务中运行，且没有统一调度或总线互斥保护，就可能并发访问同一 I2C/SPI/UART 外设。即使 HAL 能返回 busy，也会让上层状态变得不可预测。

### 2.3 状态来源过多

当前 `init`、`self_check`、`read`、`get_status` 都可以输出 `Sensor_Status_t`，函数返回值又是 `Sensor_Severity_t`。这会形成两套结果来源：

```text
函数返回值       -> Sensor_Severity_t
status 输出参数  -> severity + code + driver_error
```

如果两者不一致，注册表只能取更严重者，但仍可能拿不到具体错误码。

### 2.4 恢复判断依赖自检结果

当前应用层通过本轮自检异常 key 与告警活动表对比来判断恢复。如果异常来自采样阶段，而下一轮自检没有报异常，可能出现误清除。

### 2.5 采样状态缺少系统参考

`Sensor_Sample_t.status` 当前没有统一定义。系统没有外部参考源时，无法判断测量值“准不准”，只能判断本次采样是否存在驱动可确认的异常，例如通信失败、校验失败、没有读到数据或 GNSS 无定位。

### 2.6 设备类型字段冗余

当前运行期链路已有 `device_id`、`name`、`channel`、`value_type` 和 `unit`，足以区分设备实例、给人展示并解释采样数据。`sensor_type` 只提供粗分类，V1.0 没有明确不可替代的职责，继续保留会增加维护成本。

## 3. 调整原则

### 3.1 运行期只保留一个主动硬件访问路径

运行期主动访问硬件的入口应统一为采样链路：

```text
上层采样调度者
  -> Sensor_RegistryReadAll()
      -> driver->read()
```

`read` 负责：

- 访问硬件并读取采样数据。
- 解析原始数据并填充 `Sensor_Sample_t`。
- 根据本次访问结果刷新驱动内部状态缓存。

### 3.2 状态查询不访问硬件

`get_status` 的职责固定为读取驱动缓存。它不负责发现新错误，最新错误状态由最近一次 `init`、`self_check` 或 `read` 刷新：

```text
driver->get_status()
  -> 返回最近一次 init/self_check/read 刷新的 Sensor_Status_t
  -> 不访问 I2C/SPI/UART
  -> 不触发耗时等待
```

运行期如果 `read` 输出的任一 `Sensor_Sample_t.status == 1`，注册表再调用对应驱动的 `get_status()`，读取缓存中的 `Sensor_Status_t.code`、`severity` 和 `driver_error`，用于告警和状态汇总。

### 3.3 自检只用于启动阶段

运行期不再周期调用 `self_check` 访问硬件。上电初始化后仍需要检测关键设备是否能读到基本数据，因此将 `self_check` 收敛为启动自检语义：

```text
startup_self_check
  -> 只在初始化后的上电自检阶段调用一次
  -> 可访问硬件
  -> 只关心设备是否可访问、是否能读到基本数据
  -> 刷新驱动状态缓存
```

代码目标沿用 `self_check` 命名，降低驱动层改动范围。文档和实现必须明确：`self_check` 只表示启动自检，不表示运行期周期自检。

### 3.4 错误码仍以状态缓存为准

传感器驱动直接使用 `ERR_SENSOR_*`。注册表不再做复杂映射，只校验并透传。

```text
Sensor_Status_t.code == ERR_OK
  -> 当前设备无主错误

Sensor_Status_t.code != ERR_OK
  -> 当前设备存在异常，告警 key 使用:
     APP_MODULE_SENSOR + code + device_id
```

## 4. 目标接口

### 4.1 传感器状态

目标状态结构只保留异常判断必要字段：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Severity_t severity;
  App_ErrorCode_t code;
  uint32_t driver_error;
} Sensor_Status_t;
```

字段职责：

| 字段 | 职责 |
|---|---|
| `device_id` | 设备实例编号，作为传感器异常 `instance_id` |
| `severity` | 当前异常影响级别 |
| `code` | 当前主错误码，正常时为 `ERR_OK` |
| `driver_error` | 可选驱动细节，例如 HAL 错误、芯片状态、协议阶段码 |

### 4.2 采样结构

目标采样结构去掉 `sensor_type`，保留采样解析需要的字段：

```c
typedef struct
{
  uint16_t device_id;
  uint16_t channel;
  Sensor_ValueType_t value_type;
  uint32_t timestamp_ms;
  uint8_t status;
  char unit[SENSOR_UNIT_LEN];
  Sensor_Value_t value;
} Sensor_Sample_t;
```

字段职责：

| 字段 | 职责 |
|---|---|
| `device_id` | 设备实例编号 |
| `channel` | 同一设备的采样通道 |
| `value_type` | 说明 `value` 联合体中哪个字段有效 |
| `timestamp_ms` | 采样时间 |
| `status` | `0=正常, 1=异常` |
| `unit` | 单位字符串 |
| `value` | 采样值 |

### 4.3 驱动句柄

目标句柄建议为：

```c
typedef Sensor_Severity_t (*Sensor_InitFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_SelfCheckFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_ReadFn)(Sensor_Sample_t *samples, uint16_t max_count, uint16_t *out_count);
typedef Sensor_Severity_t (*Sensor_GetStatusFn)(Sensor_Status_t *status);

typedef struct
{
  uint16_t device_id;
  const char *name;
  Sensor_InitFn init;
  Sensor_SelfCheckFn self_check;
  Sensor_ReadFn read;
  Sensor_GetStatusFn get_status;
} Sensor_Driver_t;
```

`device_id` 用于程序定位设备，`name` 用于日志、显示和告警 payload。设备分类需求后续放入独立设备信息表，不放入运行期采样结构、状态结构或驱动句柄。

## 5. 采样状态

### 5.1 字段命名

建议保留 `Sensor_Sample_t.status` 字段名，但将其类型和语义收敛为布尔值：

```c
uint8_t status;
```

字段语义：

| 值 | 含义 |
|---:|---|
| `0` | 本次采样正常，样本可按当前业务规则使用 |
| `1` | 本次采样异常，需要通过该传感器的 `get_status()` 获取具体错误状态 |

该字段只表达本次样本是否异常，不替代 `Sensor_Status_t.code`，也不直接作为告警 key。

### 5.2 采样异常与状态查询

运行期采样流程：

```text
driver->read()
  -> 访问硬件并读取数据
  -> 发现异常时刷新驱动 cached_status
  -> 将对应样本 status 填 1

Sensor_RegistryReadAll()
  -> 发现某个样本 status == 1
  -> 调用该传感器 driver->get_status()
  -> 读取 cached_status
  -> 输出异常状态给应用层
```

`read()` 在发现异常时必须先刷新缓存状态，再返回样本。`get_status()` 只读取该缓存，不再访问硬件。

### 5.3 采样异常判断边界

V1.0 阶段不定义复杂采样质量等级。驱动只需要判断本次采样是否存在明确异常：

- 没有读到数据。
- 通信超时或设备无响应。
- 数据帧校验失败。
- 数据格式错误。
- 数据超出驱动内置的物理合理范围。
- GNSS 类设备无定位。

没有外部参考源时，不判断“测量精度高低”。

## 6. 设备分类

V1.0 运行期接口不保留 `sensor_type`。原因：

- `device_id` 已经可以唯一定位设备实例。
- `name` 可以满足日志、显示和告警 payload 的可读性。
- `channel`、`value_type` 和 `unit` 已经可以解释采样数据。
- 当前没有明确业务逻辑必须依赖 GNSS/IMU/WEATHER 这类粗分类。

后续如果 Web、日志或协议需要设备分类，新增独立设备信息表：

```c
typedef struct
{
  uint16_t device_id;
  const char *name;
  uint16_t category;
} Sensor_DeviceInfo_t;
```

该表只用于展示、筛选或协议元信息，不参与运行期驱动调用和异常判断。

## 7. 目标数据流

### 7.1 上电阶段

```text
App_TasksInit()
  -> App_RegistryRunInit()
      -> Sensor_RegistryInitAll()
          -> driver->init()
          -> 刷新驱动状态缓存

POST 自检阶段
  -> Sensor_RegistrySelfCheckAll()
      -> driver->self_check()
      -> 刷新驱动状态缓存
      -> 汇总 Sensor_Status_t
      -> 应用层投递 RAISE/CLEAR
```

### 7.2 运行阶段

```text
上层采样调度者周期采样
  -> Sensor_RegistryReadAll()
      -> driver->read()
          -> 访问硬件
          -> 输出 Sensor_Sample_t.status
          -> 刷新驱动状态缓存
      -> 发现 status==1 时调用对应驱动 get_status()
      -> 汇总异常状态
      -> 应用层投递 RAISE/CLEAR
```

说明：

- V1.0 需要一个独立的传感器采样调度入口，但该任务属于应用层实现，不属于具体驱动同事的职责。
- 具体传感器驱动只实现 `Sensor_Driver_t`，不创建任务、不投递告警、不组装 CNS 帧。

### 7.3 采样数据快照

多个传感器和多通道数据通过 `Sensor_Sample_t` 数组向上层传递。运行期建议由采样调度者维护一份最近采样快照：

```c
#define SENSOR_SAMPLE_CACHE_MAX 16U

typedef struct
{
  uint32_t update_ms;
  uint16_t count;
  Sensor_Sample_t samples[SENSOR_SAMPLE_CACHE_MAX];
} Sensor_Snapshot_t;
```

建议服务接口：

```c
Sensor_Severity_t Sensor_ServiceUpdate(void);
Sensor_Severity_t Sensor_ServiceGetSnapshot(Sensor_Snapshot_t *snapshot);
```

数据流：

```text
上层采样调度者
  -> Sensor_ServiceUpdate()
      -> Sensor_RegistryReadAll()
      -> 写入 Sensor_Snapshot 缓存

ProtocolTask
  -> Sensor_ServiceGetSnapshot()
  -> 读取最近一次 Sensor_Snapshot
  -> 构造 CNS_MSG_SENSOR 数据帧
  -> 交给 CommTask 发送
```

`ProtocolTask` 不直接调用具体传感器驱动，也不直接访问硬件。多任务并发访问快照时，`Sensor_ServiceGetSnapshot()` 内部使用互斥锁、临界区或双缓冲保证快照一致。

`Sensor_Sample_t` 能承载多传感器和多通道数据：

```text
device_id  -> 设备实例
channel    -> 同一设备内的采样项
value_type -> value 联合体有效字段
unit       -> 单位
status     -> 本条样本是否异常
```

多通道传感器的 `read()` 必须按完整采样组输出。驱动在写入 `samples[]` 前先判断本次完整输出需要的样本数量：

```text
required_count > max_count
  -> 不写入 samples[]
  -> out_count = 0
  -> 刷新状态缓存
  -> 返回异常

required_count <= max_count
  -> 连续写入 samples[0..required_count-1]
  -> out_count = required_count
```

V1.0 不输出半包，避免 Protocol、日志或显示拿到同一个设备不完整的一组采样。

如果后续要求每一条采样都不能丢失，再从快照模型升级为 `sample batch queue`。V1.0 先采用快照更简单，也更适合状态显示、日志和周期上报。

### 7.4 状态汇总阶段

```text
CNS_State 汇总 / 日志 / HMI / 上报
  -> Sensor_RegistryGetStatusAll()
      -> driver->get_status()
      -> 只读缓存，不访问硬件
```

## 8. 总线访问策略

本次传感器接口调整不直接实现总线驱动，但需要明确约束：

- 只有 `init`、`self_check`、`read` 能访问硬件。
- `get_status` 不访问硬件。
- 运行期由一个上层采样调度者统一调用 `Sensor_RegistryReadAll()`。
- 如果后续存在多个任务访问同一总线，必须在 BSP 或 Bus 层提供互斥锁。
- 不允许 HMI、日志、协议任务直接调用具体驱动访问硬件。

## 9. 告警恢复策略

恢复判断不再依赖单独的周期自检结果，而是依赖最近一次采样或启动探测刷新后的驱动状态缓存。

每轮采样或探测后，注册表根据异常样本或异常探测结果获取当前异常 key 集合：

```text
APP_MODULE_SENSOR + Sensor_Status_t.code + Sensor_Status_t.device_id
```

应用层处理：

```text
当前异常 key 存在，活动表不存在 -> RAISE
当前异常 key 存在，活动表已存在 -> 更新 last_tick_ms
活动表存在，当前异常 key 不存在 -> CLEAR
```

如果异常数量超过输出缓冲区容量，不执行恢复清除，避免误清除仍然存在但未被写入缓冲区的异常。

## 10. 影响范围

| 文件 | 调整内容 |
|---|---|
| `Sensor_Driver/Inc/sensor_driver.h` | 调整句柄接口语义，移除 `sensor_type`，明确 `get_status` 只读缓存 |
| `Sensor_Driver/Inc/sensor_data.h` | 移除 `sensor_type`，将 `status` 类型和语义收敛为布尔值 |
| `Sensor_Driver/Src/sensor_registry.c` | 运行期状态来源改为 `read` 后的缓存状态；采样异常时调用对应驱动 `get_status()` |
| `Sensor_Driver/Src/mock_sensor.c` | 按缓存状态模型改造 mock 驱动 |
| `Sensor_Driver/Inc/sensor_service.h` | 建议新增采样服务接口和 `Sensor_Snapshot_t` |
| `Sensor_Driver/Src/sensor_service.c` | 建议新增最近采样快照缓存，供 Protocol 读取 |
| `App/Src/app_registry.c` | 传感器异常触发/恢复改为读取注册表当前状态集合 |
| `Docs/传感器驱动接入指南.md` | 更新驱动开发规范 |
| `Docs/通信数据结构设计说明.md` | 同步传感器接口、采样状态和设备分类说明 |
| `Docs/protocol/CNS帧格式定义.md` | 同步 SENSOR payload 中 `status` 布尔语义 |

## 11. 建议执行顺序

1. 保留驱动句柄中的 `self_check` 命名，但将语义收敛为启动自检。
2. 简化 `Sensor_Status_t`，移除 `sensor_type`，只保留 `device_id/severity/code/driver_error`。
3. 简化 `Sensor_Sample_t`，移除 `sensor_type`，将 `status` 改为 `uint8_t` 布尔语义。
4. 改造 mock 驱动，增加内部缓存状态。
5. 改造传感器注册表，运行期以 `read` 刷新状态。
6. 新增或调整传感器采样服务，维护 `Sensor_Snapshot_t`。
7. 改造应用注册表的传感器异常触发和恢复逻辑。
8. 更新传感器接入指南、通信数据结构说明和 CNS 帧格式文档。
9. 编译并运行现有单元测试。
10. 执行 Keil build verification。

## 12. 结论

传感器驱动层应把运行期硬件访问集中到采样链路。自检只在初始化后执行一次，用于确认设备能否读到基本数据。运行期由 `read()` 发现异常并刷新缓存状态，`get_status()` 只读取缓存。这样可以减少总线访问次数，降低并发访问风险，并让异常触发和恢复判断依赖同一份状态事实。
