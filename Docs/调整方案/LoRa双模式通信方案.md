# LoRa 双模式通信方案

> 文档状态：待会议评审。本文仅用于讨论 V1.0 主控箱的 LoRa 链路工作方式，不代表已经确定的产品约束，评审通过前不得直接据此实施。

## 1. 背景

V1.0 主控箱既需要作为一个能够独立运行的产品，也需要作为“一中心、五平台”架构中的通信节点。

主控箱独立运行时，本地传感器采集、显示、日志、告警和控制功能不依赖中心或上位机。LoRa 只承担对外通信，不应成为本地功能闭环的前置条件。

LoRa 通常工作在半双工模式，同一节点不能同时发送和接收。如果主控箱按固定周期主动上报，而上位机可以在任意时刻下发控制命令，可能出现以下冲突：

```text
主控箱正在发送周期数据
        ×
中心同时发送控制命令
        ↓
双方均无法正确接收，命令或上报帧丢失
```

本地互斥锁只能避免主控箱内部多个任务同时操作 LoRa，不能协调多个无线节点对空口的占用。因此必须从链路访问策略上解决冲突。

## 2. 设计目标

- 主控箱脱离中心后仍能完成全部本地功能。
- 单机展示或点对点连接时，主控箱能够主动上报状态和采样数据。
- 正式组网时，由中心统一调度五个平台，避免多个节点主动发送造成碰撞。
- 控制命令具备 ACK、超时、重试和重复命令保护。
- 通信模式明确可配置，避免运行期自动切换引起网络振荡。
- 两种模式复用相同的 CNS 帧格式、解析流程和业务命令处理链路。

## 3. 基本原则

### 3.1 产品运行与链路模式分离

主控箱的本地功能始终运行：

```text
传感器采集 → 状态汇总 → 本地显示
                     ├→ SD 日志
                     ├→ 本地告警
                     └→ 本地控制
```

LoRa 链路模式只决定 CNS 数据何时发送以及何时接收，不控制本地采集、显示、日志和告警任务是否运行。

### 3.2 半双工由 Comm 模块统一调度

只有 Comm 模块能够切换 LoRa 收发状态。Protocol、Command、Acquisition 等模块只通过队列提交待发送帧或接收解析结果，不直接操作 E22。

发送互斥锁可以保护本地 E22 操作，但空口冲突仍由主动上报策略、接收窗口或中心轮询机制解决。

### 3.3 不建议 V1.0 自动切换模式

不建议根据“是否收到中心心跳”在主动上报和被动轮询之间自动切换。中心短时离线后，五个平台可能同时转为主动上报，造成集中碰撞；中心恢复后，各节点切换时机不同，也可能产生模式振荡。

V1.0 建议采用明确配置的模式，配置修改后持久化，并在下次启动或链路安全重启后生效。

## 4. 候选通信模式

### 4.1 主动上报模式

适用场景：

- 主控箱作为独立产品展示。
- 单主控箱与单上位机点对点连接。
- 没有持续在线的中心调度器。

主控箱按固定周期主动发送，发送结束后开放接收窗口：

```text
等待上报周期
    ↓
发送 STATUS/SENSOR/ALARM
    ↓
等待 E22 发送完成并切换为接收
    ↓
保持接收窗口
    ├→ 收到 QUERY/CONFIG/CONTROL：解析、执行、回复 ACK
    └→ 窗口超时：返回等待上报周期
```

约束：

- 上位机不能假设主控箱始终处于接收状态，应在主控箱上报后或约定接收窗口内下发命令。
- 上报帧应携带 `ACK_REQ` 时，主控箱需要在等待 ACK 期间保持接收。
- 告警帧可以进入高优先级发送队列，但是否允许立即打断普通上报周期需要会议确认。
- 多个主动节点使用同一信道时仍存在碰撞风险，因此该模式不作为五平台正式组网的推荐模式。

### 4.2 被动轮询模式

适用场景：

- 一个中心统一管理五个平台。
- 上位机或中心持续在线并承担链路调度。
- 需要确定性的控制命令响应和节点访问顺序。

中心拥有空口调度权，主控箱通常保持接收，不主动发送普通遥测数据：

```text
中心 → 平台 1：POLL，可携带 QUERY/CONFIG/CONTROL
平台 1 → 中心：ACK + 待上报数据

中心 → 平台 2：POLL，可携带 QUERY/CONFIG/CONTROL
平台 2 → 中心：ACK + 待上报数据

……依次轮询其他平台
```

主控箱状态流程：

```text
持续接收
  ↓ 收到发给本机的帧
解析并投递命令
  ↓
等待命令执行结果
  ↓
优先回复 ACK/命令结果
  ↓
按剩余响应窗口发送状态、告警或采样数据
  ↓
返回接收
```

约束：

- 单个平台响应超时不能阻塞中心对后续平台的轮询。
- 主控箱不主动发送普通 STATUS/SENSOR，数据先在本地更新，收到轮询后再上报最新快照。
- 严重告警是否允许主控箱脱离轮询主动抢占信道需要会议确认。V1.0 的保守方案是不抢占，随下一次轮询上报。

## 5. 建议配置模型

候选配置如下，最终字段和默认值待评审：

```c
typedef enum {
  APP_COMM_MODE_ACTIVE_REPORT = 0,
  APP_COMM_MODE_POLLED
} App_CommMode_t;

typedef struct {
  App_CommMode_t mode;
  uint32_t report_interval_ms;
  uint32_t receive_window_ms;
  uint32_t response_timeout_ms;
  uint8_t retry_count;
} App_CommConfig_t;
```

配置来源候选：

- 出厂默认配置。
- 本地 HMI 设置。
- 调试串口设置。
- CNS `CONFIG` 指令远程设置。

通信模式和关键参数需要持久化。远程修改模式时，应先返回 ACK，再在当前事务结束后切换，避免在应答发送过程中改变收发策略。

## 6. Comm 状态机

Comm 模块应同时管理链路模式和 E22 半双工收发状态，禁止上层任务直接切换 E22 工作状态。

候选状态：

```c
typedef enum {
  APP_COMM_STATE_INIT = 0,
  APP_COMM_STATE_RX_IDLE,
  APP_COMM_STATE_TX_WAIT,
  APP_COMM_STATE_TX_ACTIVE,
  APP_COMM_STATE_RX_WINDOW,
  APP_COMM_STATE_WAIT_RESPONSE,
  APP_COMM_STATE_RECOVERY
} App_CommState_t;
```

模式之间复用发送、接收、超时和重试实现：

| 状态 | 主动上报模式 | 被动轮询模式 |
|---|---|---|
| `RX_IDLE` | 等待上报周期或接收窗口内命令 | 持续等待中心轮询或命令 |
| `TX_WAIT` | 到达上报周期或存在高优先级帧 | 收到轮询后准备响应 |
| `TX_ACTIVE` | 发送主动上报或 ACK | 发送轮询响应或 ACK |
| `RX_WINDOW` | 上报后开放命令接收窗口 | 一般不单独使用 |
| `WAIT_RESPONSE` | 等待需要的 ACK | 等待命令执行结果或中心确认 |
| `RECOVERY` | 超时、E22 异常后的恢复 | 超时、E22 异常后的恢复 |

E22 发送完成应优先通过 `AUX` 引脚或明确的模块状态判断，不应仅使用固定延时估算。收发切换需要保留保护时间，具体时间由 E22 型号、空中速率和工作模式确定。

## 7. 发送优先级

建议 Comm 内部至少区分两级发送队列：

| 优先级 | 数据类型 | 说明 |
|---|---|---|
| 高 | ACK、控制执行结果、链路管理帧 | 优先保证控制闭环 |
| 普通 | STATUS、SENSOR、心跳、普通日志状态 | 可以合并、覆盖旧快照或延后发送 |

告警帧是否属于最高优先级，以及在被动轮询模式下能否主动发送，需要会议确认。

普通周期数据不应无限积压。队列满时应优先保留最新状态快照，允许丢弃已经过期的旧 SENSOR/STATUS 帧；ACK 和控制执行结果不能被普通遥测挤占。

## 8. 命令可靠性

两种模式都需要复用以下机制：

- `source + sequence` 用于识别请求。
- 主控箱缓存最近执行过的控制命令序号，收到重传帧时不重复执行有副作用的命令，只重发上次结果。
- 请求携带 `ACK_REQ` 时，接收方必须在约定超时内返回 ACK。
- 发送方超时后执行有限次数重试，并设置 `IS_RETRY`。
- 超过重试次数后记录通信异常和丢包统计，不无限阻塞 Comm 任务。
- 急停等安全动作不能只依赖 LoRa，应保留本地急停入口。

## 9. 对现有设计的影响

### 9.1 应用层

- `App_CommTask` 从简单收发任务变为链路状态机和唯一 E22 所有者。
- Protocol 负责 CNS 编解码，不负责决定空口发送时机。
- Command 负责执行命令，不直接发送 ACK，由 Protocol 编码后交给 Comm。
- 应用层需要提供最新状态和采样快照，避免被动轮询模式下堆积历史数据帧。

### 9.2 CNS 协议

- 现有 `ACK_REQ`、`IS_ACK`、`IS_RETRY` 和 `sequence` 可以继续使用。
- 需要确认是否新增专用 `POLL` 消息类型，或使用现有 `QUERY` 表达轮询。
- 需要补充主动模式接收窗口、轮询响应窗口、超时和重试规则。
- 需要定义通信模式对应的 `CONFIG param_id`。

### 9.3 E22/BSP 层

- 提供明确的发送完成、接收数据和工作状态切换接口。
- 明确 `AUX`、M0、M1 等引脚的使用方式。
- 串口 DMA/中断、帧接收缓存和超时机制需要与 Comm 状态机配合。

### 9.4 上位机和中心

- 主动模式下，上位机需要识别主控箱上报并在接收窗口内下发命令。
- 被动模式下，中心需要实现五个平台的轮询调度、节点超时隔离和重试。
- 上位机需要能够读取和配置当前通信模式。

### 9.5 测试

- 主动上报周期和接收窗口验证。
- 上报与命令同时发生时的冲突恢复。
- 被动轮询下五个平台依次响应。
- 单节点离线不影响其他节点。
- ACK 丢失、命令重传和重复命令保护。
- 模式配置持久化和重启恢复。
- E22 发送完成、收发切换和异常恢复。

## 10. 会议决策项

以下内容需要评审后才能形成正式约束：

1. V1.0 是否正式支持主动上报和被动轮询两种模式。
2. 出厂默认模式是什么，不同交付场景是否允许使用不同默认值。
3. 模式通过 HMI、串口和远程配置中的哪些入口修改。
4. 主动模式的默认上报周期、接收窗口和重试次数。
5. 被动模式是否新增 `CNS_MSG_POLL`，还是复用 `CNS_MSG_QUERY`。
6. 中心轮询五个平台的周期、单节点响应窗口和超时时间。
7. 被动轮询模式下，严重告警是否允许主动抢占信道。
8. 告警、ACK、控制应答和普通数据的发送优先级。
9. E22 的具体型号、空中速率、透明传输或定点传输方式。
10. 通信模式切换是立即生效、事务结束后生效，还是重启后生效。

## 11. 当前建议

- 保留两种显式配置的通信模式，使主控箱同时适用于独立产品和正式组网。
- 单机或点对点场景使用主动上报模式，发送后开放接收窗口。
- 一中心五平台场景使用被动轮询模式，由中心拥有空口调度权。
- V1.0 不加入运行期自动模式判断和自动切换。
- 评审完成前，不修改现有应用层任务方案、CNS 正式协议和 Comm 接口。
