# 通信数据结构设计说明

本文整理当前工程中与通信相关的数据结构，以及后续接入传感器、无线通信设备和 Linux 网关时建议补充的定义位置。

核心原则：

- `Bsp` 只处理 UART、RS485、CAN、I2C、SPI 等底层收发，不理解业务数据。
- `Sensor_Driver` 负责把传感器原始数据解析成统一采样结构。
- `App/Protocol` 负责把系统状态、传感器数据、告警和控制反馈封装成主控对外通信帧。
- 传感器和主控之间的数据结构，不与主控和网关之间的数据结构混用。
- `CNS_State` 是应用层汇总快照，用于日志记录和对外上报；驱动和底层模块不得直接写入 `CNS_State`。

## 1. 通信链路划分

```text
传感器/有线设备
  -> Bsp raw transfer
  -> Sensor_Driver parse
  -> Sensor_Sample_t
  -> Sensor_Snapshot_t
  -> App logger/display/protocol

主控箱 STM32F407
  -> App 聚合 CNS_State 快照
  -> CNS_Frame_t
  -> Comm/Link driver
  -> Linux 网关 / 无线通信设备 / 上位机
```

## 2. 传感器到主控的数据结构

### 2.1 底层有线接口定义位置

当前定义位置：

```text
Bsp/Inc/bsp_uart.h
Bsp/Inc/bsp_can.h
Bsp/Inc/bsp_i2c.h
Bsp/Inc/bsp_spi.h
Bsp/Inc/bsp_sd.h
```

当前 BSP 仍是占位接口。后续建议新增统一总线定义：

```text
Bsp/Inc/bsp_bus.h
```

建议结构：

```c
typedef enum {
  BSP_BUS_UART = 0,  /* 普通串口，适合 GNSS、DWIN、透明数传等 */
  BSP_BUS_RS485,     /* 半双工总线，适合长线传感器或多节点设备 */
  BSP_BUS_CAN,       /* CAN 总线，适合抗干扰设备、Remote ID 或电源节点 */
  BSP_BUS_I2C,       /* 板内低速传感器，例如 BME280、MPU6050 */
  BSP_BUS_SPI        /* 板内高速外设或 SPI 模式 SD 卡 */
} Bsp_BusType_t;

typedef struct
{
  Bsp_BusType_t type;     /* 总线类型 */
  uint8_t port;           /* 端口编号，例如 UART1/CAN1/I2C1 */
  uint32_t timeout_ms;    /* 单次收发超时时间 */
} Bsp_BusHandle_t;

typedef struct
{
  uint8_t *data;          /* 原始数据缓冲区 */
  uint16_t length;        /* 当前有效字节数 */
  uint32_t timestamp_ms;  /* 接收或发送时刻，单位 ms */
} Bsp_RawFrame_t;
```

BSP 层只返回原始字节、传输状态和超时状态，不直接返回 GNSS 坐标、气象数值、Remote ID 状态等业务数据。

### 2.2 传感器采样结构

当前定义位置：

```text
Sensor_Driver/Inc/sensor_data.h
```

目标接口：

```c
#define SENSOR_UNIT_LEN 8U

typedef enum {
  SENSOR_VALUE_NONE = 0,    /* 无有效数据 */
  SENSOR_VALUE_FLOAT,       /* 单精度浮点值 */
  SENSOR_VALUE_INT32,       /* 有符号整型值 */
  SENSOR_VALUE_UINT32,      /* 无符号整型值 */
  SENSOR_VALUE_BOOL,        /* 布尔或开关量 */
  SENSOR_VALUE_GEO          /* 经纬高数据 */
} Sensor_ValueType_t;

typedef struct
{
  int32_t latitude_deg_e7;   /* 纬度，单位 degree * 1e7 */
  int32_t longitude_deg_e7;  /* 经度，单位 degree * 1e7 */
  int32_t altitude_mm;       /* 高度，单位 mm */
} Sensor_GeoValue_t;

typedef union {
  float f32;                 /* 浮点采样值 */
  int32_t i32;               /* 有符号整型采样值 */
  uint32_t u32;              /* 无符号整型采样值 */
  uint8_t boolean;           /* 开关量 */
  Sensor_GeoValue_t geo;     /* 经纬高组合值 */
} Sensor_Value_t;

typedef struct
{
  uint16_t device_id;              /* 设备编号，由项目接口表统一分配 */
  uint16_t channel;                /* 通道号，例如电源第 1 路、电机第 2 路 */
  Sensor_ValueType_t value_type;   /* value 联合体中当前有效的数据类型 */
  uint32_t timestamp_ms;           /* 采样时间戳，单位 ms */
  uint8_t status;                  /* 0=正常，1=异常 */
  char unit[SENSOR_UNIT_LEN];      /* 单位，例如 "V"、"m/s"、"deg" */
  Sensor_Value_t value;            /* 采样值 */
} Sensor_Sample_t;
```

使用约束：

- 一个传感器可以一次返回多个 `Sensor_Sample_t`，例如气象模块同时返回温度、湿度、气压。
- `device_id` 必须来自统一接口表，不能由驱动随意硬编码冲突编号。
- `channel` 用于区分同一设备输出的多个采样项。
- `status` 只表达本条样本是否异常，不替代 `Sensor_Status_t.code`。
- 多通道传感器必须按完整采样组输出；如果调用方提供的 `max_count` 不足，驱动不输出半包，`out_count` 保持 `0U` 并返回异常。
- 经纬高使用整数定点表示，避免在协议传输中依赖 double。
- V1.0 运行期结构不保留 `sensor_type`，设备分类后续如有需要由独立设备信息表维护。

### 2.3 传感器驱动接口

当前定义位置：

```text
Sensor_Driver/Inc/sensor_driver.h
```

目标接口：

```c
typedef enum {
  SENSOR_SEVERITY_NORMAL = 0, /* 正常 */
  SENSOR_SEVERITY_GENERAL,    /* 一般异常，不影响主系统运行 */
  SENSOR_SEVERITY_IMPORTANT,  /* 重要异常，影响实验或数据可信度 */
  SENSOR_SEVERITY_CRITICAL    /* 严重异常，需要停机或人工处理 */
} Sensor_Severity_t;

typedef struct
{
  uint16_t device_id;              /* 设备编号 */
  Sensor_Severity_t severity;      /* 异常严重程度 */
  App_ErrorCode_t code;            /* 统一传感器错误码，正常时为 ERR_OK */
  uint32_t driver_error;           /* 可选驱动内部细节码 */
} Sensor_Status_t;

typedef Sensor_Severity_t (*Sensor_InitFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_SelfCheckFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_ReadFn)(Sensor_Sample_t *samples, uint16_t max_count, uint16_t *out_count);
typedef Sensor_Severity_t (*Sensor_GetStatusFn)(Sensor_Status_t *status);

typedef struct
{
  uint16_t device_id;              /* 设备编号 */
  const char *name;                /* 设备名称，日志和调试使用 */
  Sensor_InitFn init;              /* 驱动初始化 */
  Sensor_SelfCheckFn self_check;   /* 启动自检 */
  Sensor_ReadFn read;              /* 读取统一采样数据 */
  Sensor_GetStatusFn get_status;   /* 查询缓存状态 */
} Sensor_Driver_t;
```

传感器驱动只需要实现 `Sensor_Driver_t`，不直接操作日志、显示、告警或网关协议。驱动返回 `Sensor_Severity_t` 表达影响级别，通过 `Sensor_Status_t.code` 直接输出统一 `ERR_SENSOR_*` 错误码；`driver_error` 只保存 HAL 返回值、芯片状态或驱动阶段码等可选调试细节。

运行期硬件访问集中在 `read`。`self_check` 只在 `init` 之后的上电自检阶段调用，不用于运行期周期自检；`get_status` 只返回最近一次 `init/self_check/read` 刷新的缓存状态，不访问硬件。

### 2.4 传感器注册表

当前定义位置：

```text
Sensor_Driver/Inc/sensor_registry.h
Sensor_Driver/Src/sensor_registry.c
```

目标入口：

```c
Sensor_Severity_t Sensor_RegistryInitAll(void);
Sensor_Severity_t Sensor_RegistrySelfCheckAll(Sensor_Status_t *status_list, uint16_t max_count, uint16_t *out_count);
Sensor_Severity_t Sensor_RegistryReadAll(Sensor_Sample_t *samples,
                                         uint16_t max_sample_count,
                                         uint16_t *out_sample_count,
                                         Sensor_Status_t *status_list,
                                         uint16_t max_status_count,
                                         uint16_t *out_status_count);
Sensor_Severity_t Sensor_RegistryGetStatusAll(Sensor_Status_t *status_list, uint16_t max_count, uint16_t *out_count);
uint16_t Sensor_RegistryGetCount(void);
const Sensor_Driver_t *Sensor_RegistryGetDriver(uint16_t index);
```

`Sensor_RegistrySelfCheckAll()` 只用于上电自检阶段，确认设备是否可访问并刷新状态缓存，不用于运行期周期自检。`Sensor_RegistryReadAll()` 用于运行期采样，在输出采样数据的同时输出读取过程中的异常状态明细。`Sensor_RegistryGetStatusAll()` 只读取缓存状态，不访问硬件。

后续新增具体传感器时，把驱动句柄挂入 `s_sensor_drivers[]`：

```c
static const Sensor_Driver_t *const s_sensor_drivers[] =
{
  &g_gnss_driver,
  &g_weather_driver,
  &g_power_driver
};
```

### 3.1 当前协议结构


当前已实现的是 demo 级协议结构：


### 3.2 建议新增的正式协议定义位置

建议把正式 CNS 协议从 `app_protocol.h` 中拆出，保持 `app_protocol.h` 作为任务入口和协议模块门面。


### 3.4 正式 CNS 通信帧

建议定义位置：

```text
App/Inc/cns_frame.h
```

建议结构：

```c
#define CNS_FRAME_HEADER_0 0xAAU
#define CNS_FRAME_HEADER_1 0x55U
#define CNS_FRAME_VERSION  0x01U
#define CNS_FRAME_PAYLOAD_MAX_LEN 128U

typedef struct
{
  uint8_t header0;                         /* 固定 0xAA */
  uint8_t header1;                         /* 固定 0x55 */
  uint8_t version;                         /* 协议版本 */
  CNS_MessageType_t msg_type;              /* 消息类型 */
  uint16_t sequence;                       /* 帧序号，用于去重和应答匹配 */
  uint8_t source;                          /* 源节点，取值见 CNS_NodeType_t */
  uint8_t target;                          /* 目标节点，取值见 CNS_NodeType_t */
  uint32_t timestamp_ms;                   /* 主控本地时间戳 */
  uint16_t payload_len;                    /* payload 有效长度 */
  uint8_t payload[CNS_FRAME_PAYLOAD_MAX_LEN]; /* 负载数据 */
  uint16_t checksum;                       /* 校验值，建议 CRC16 */
} CNS_Frame_t;
```

建议编解码接口：

```c
typedef enum {
  CNS_FRAME_OK = 0,
  CNS_FRAME_INVALID_PARAM,
  CNS_FRAME_INVALID_LENGTH,
  CNS_FRAME_INVALID_HEADER,
  CNS_FRAME_INVALID_VERSION,
  CNS_FRAME_INVALID_CHECKSUM
} CNS_FrameResult_t;

CNS_FrameResult_t CNS_EncodeFrame(const CNS_Frame_t *frame,
                                  uint8_t *buffer,
                                  uint16_t buffer_size,
                                  uint16_t *out_len);

CNS_FrameResult_t CNS_DecodeFrame(const uint8_t *buffer,
                                  uint16_t length,
                                  CNS_Frame_t *frame);
```

### 3.5 主控对网关的负载来源

主控和网关之间统一发送 `CNS_Frame_t`，但负载来源可以不同：

| 负载来源 | 当前定义位置 | 对外消息类型 |
| --- | --- | --- |
| 传感器采样 | `Sensor_Driver/Inc/sensor_data.h` 的 `Sensor_Sample_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_SENSOR_DATA` |
| 系统模块状态 | `App/Inc/app_status.h` 的 `App_ModuleStatus_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_STATUS` |
| 告警状态 | `App/Inc/app_alarm.h` 的 `App_AlarmRecord_t`，对外错误码使用 `App/Inc/app_error.h` 的 `App_ErrorCode_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_ALARM` |
| 心跳 | `App/Protocol` 后续定义 | `CNS_MSG_HEARTBEAT` |
| 参数配置 | 后续 `App/Inc/app_config.h` | `CNS_MSG_CONFIG` |
| 控制命令 | 后续 `App/Inc/cns_message.h` | `CNS_MSG_CONTROL` |

## 4. 系统状态和告警结构

### 4.1 系统模块状态

当前定义位置：

```text
App/Inc/app_status.h
```

当前已实现：

```c
typedef enum {
  APP_MODULE_SYSTEM = 0,    /* 系统主任务 */
  APP_MODULE_COMM,          /* 通信模块 */
  APP_MODULE_PROTOCOL,      /* 协议模块 */
  APP_MODULE_LOGGER,        /* 日志模块 */
  APP_MODULE_DISPLAY,       /* 显示模块 */
  APP_MODULE_ALARM,         /* 告警模块 */
  APP_MODULE_STORAGE,       /* 存储模块 */
  APP_MODULE_SENSOR,        /* 传感器模块 */
  APP_MODULE_COUNT
} App_ModuleId_t;

typedef enum {
  APP_STATE_INIT = 0,       /* 初始化中 */
  APP_STATE_OK,             /* 正常 */
  APP_STATE_WARNING,        /* 警告 */
  APP_STATE_ERROR,          /* 错误 */
  APP_STATE_TIMEOUT,        /* 超时 */
  APP_STATE_OFFLINE         /* 离线 */
} App_ModuleState_t;

typedef struct
{
  App_ModuleState_t state;  /* 当前状态 */
  uint32_t heartbeat;       /* 心跳计数 */
  uint32_t last_tick_ms;    /* 最近状态更新时间 */
  App_ErrorCode_t code;     /* 标准错误码 */
} App_ModuleStatus_t;
```

`App_ModuleStatus_t` 只表达系统健康状态，不承载传感器采样值。

### 4.2 应用层汇总状态

`CNS_State` 是应用层根据传感器采样、系统模块状态、告警状态、LoRa 链路统计和参数状态生成的汇总快照。各模块只更新自身状态或通过统一接口输出数据，不直接写入 `CNS_State`。

日志、HMI 刷新和对外协议上报应读取同一份 `CNS_State` 快照。多任务访问时应采用互斥锁、临界区、消息队列或双缓冲快照，避免日志和上报读到不一致的数据组合。

### 4.3 统一异常消息和错误码

当前定义位置：

```text
App/Inc/app_alarm.h
App/Inc/app_error.h
```

当前告警链路不再维护独立 `App_AlarmId_t` 和 active mask。普通模块统一构造 `App_AlarmMsg_t`，通过 `App_AlarmPost()` 投递到告警队列；告警任务按 `source + code + instance_id` 更新活动异常记录表。

告警运行时统一采用队列入口：

```text
模块 -> 构造 App_AlarmMsg_t -> App_AlarmPost() -> 告警队列 -> App_AlarmTask() -> App_AlarmRecord_t
```

`App_AlarmMsg_t.header.code` 是标准错误码，payload 只保存现场上下文。活动异常唯一 key 为：

```text
source + code + instance_id
```

当前错误码采用模块化 `ERR_` 命名，详见 `Docs/protocol/错误码表.md`。

```c
typedef enum {
  ERR_OK = 0x0000U,
  ERR_SYSTEM_SELF_CHECK_FAILED = 0x0101U,
  ERR_SENSOR_OFFLINE = 0x0201U,
  ERR_SENSOR_NO_FIX = 0x0204U,
  ERR_COMM_LORA_OFFLINE = 0x0301U,
  ERR_DISPLAY_OFFLINE = 0x0401U,
  ERR_STORAGE_NOT_READY = 0x0501U,
  ERR_PROTOCOL_PARSE_FAILED = 0x0601U,
  ERR_UNKNOWN = 0x0FFFU
} App_ErrorCode_t;
```

告警记录：

```c
typedef struct
{
  uint8_t active;
  App_ModuleId_t source;
  App_AlarmSeverity_t severity;
  App_ErrorCode_t code;
  App_AlarmPayloadType_t payload_type;
  uint16_t instance_id;
  uint32_t first_tick_ms;
  uint32_t last_tick_ms;
  App_AlarmPayload_t payload;
} App_AlarmRecord_t;
```

`CNS_State.alarm.current_code`、HMI `alarm_code`、CSV `alarm_code` 和对外上报统一使用 `App_ErrorCode_t`，不使用告警项或 active mask。

## 5. 后续建议新增目录

当前已有：

```text
Bsp/
Sensor_Driver/
App/
```

后续接入网关或无线通信设备时，建议新增平级通信链路层：

```text
Comm_Driver/
  Inc/comm_link.h
  Inc/comm_link_registry.h
  Src/comm_link_registry.c
```

建议接口：

```c
typedef enum {
  COMM_LINK_UART_GATEWAY = 0,  /* 连接 Linux 网关的 UART */
  COMM_LINK_LORA,              /* E22 LoRa 数传 */
  COMM_LINK_CAN_GATEWAY,       /* CAN 网关或 CAN 设备 */
  COMM_LINK_MESH_STATUS        /* Mesh 模块状态链路 */
} Comm_LinkType_t;

typedef enum {
  COMM_LINK_OK = 0,
  COMM_LINK_WARNING,
  COMM_LINK_NOT_READY,
  COMM_LINK_ERROR
} Comm_LinkResult_t;

typedef struct
{
  uint8_t link_id;       /* 链路编号 */
  Comm_LinkType_t type;  /* 链路类型 */
  const char *name;      /* 链路名称 */
  Comm_LinkResult_t (*init)(void);
  Comm_LinkResult_t (*self_check)(uint32_t *error_code);
  Comm_LinkResult_t (*send)(const uint8_t *data, uint16_t length);
  Comm_LinkResult_t (*receive)(uint8_t *data, uint16_t max_len, uint16_t *out_len);
} Comm_LinkDriver_t;
```

`Comm_Driver` 负责选择链路和收发字节；`App/Protocol` 负责 `CNS_Frame_t` 编解码。两者不要混在一个模块里。

## 7. 迁移和扩展约束

- 具体传感器驱动不得直接调用显示、日志、协议发送接口。
- 传感器采样数据统一通过 `Sensor_Sample_t` 输出。
- 系统健康状态统一通过 `App_ModuleStatus_t` 输出。
- 主控对外通信统一通过 `CNS_Frame_t` 输出。
- HAL 相关句柄和寄存器操作应限制在 `Bsp` 内部。
- 网关、LoRa、Mesh 等链路应放入后续 `Comm_Driver`，不应写入 `Sensor_Driver`。
