# 通信数据结构设计说明

本文整理当前工程中与通信相关的数据结构，以及后续接入传感器、无线通信设备和 Linux 网关时建议补充的定义位置。

核心原则：

- `Bsp` 只处理 UART、RS485、CAN、I2C、SPI 等底层收发，不理解业务数据。
- `Sensor_Driver` 负责把传感器原始数据解析成统一采样结构。
- `App/Protocol` 负责把系统状态、传感器数据、告警和控制反馈封装成主控对外通信帧。
- 传感器和主控之间的数据结构，不与主控和网关之间的数据结构混用。
- `CNS_State` 是应用层汇总快照，用于日志记录和对外上报；驱动和底层模块不得直接写入 `CNS_State`。

## 1. 通信链路划分

```text
传感器/有线设备
  -> Bsp raw transfer
  -> Sensor_Driver parse
  -> Sensor_Sample_t
  -> App logger/display/protocol

主控箱 STM32F407
  -> App 聚合 CNS_State 快照
  -> CNS_Frame_t
  -> Comm/Link driver
  -> Linux 网关 / 无线通信设备 / 上位机
```

## 2. 传感器到主控的数据结构

### 2.1 底层有线接口定义位置

当前定义位置：

```text
Bsp/Inc/bsp_uart.h
Bsp/Inc/bsp_can.h
Bsp/Inc/bsp_i2c.h
Bsp/Inc/bsp_spi.h

```

当前 BSP 仍是占位接口。后续建议新增统一总线定义：

```text
Bsp/Inc/bsp_bus.h
```

建议结构：

```c
typedef enum {
  BSP_BUS_UART = 0,  /* 普通串口，适合 GNSS、DWIN、透明数传等 */
  BSP_BUS_RS485,     /* 半双工总线，适合长线传感器或多节点设备 */
  BSP_BUS_CAN,       /* CAN 总线，适合抗干扰设备、Remote ID 或电源节点 */
  BSP_BUS_I2C,       /* 板内低速传感器，例如 BME280、MPU6050 */
  BSP_BUS_SPI        /* 板内高速外设或 SPI 模式 SD 卡 */
} Bsp_BusType_t;

typedef struct
{
  Bsp_BusType_t type;     /* 总线类型 */
  uint8_t port;           /* 端口编号，例如 UART1/CAN1/I2C1 */
  uint32_t timeout_ms;    /* 单次收发超时时间 */
} Bsp_BusHandle_t;

typedef struct
{
  uint8_t *data;          /* 原始数据缓冲区 */
  uint16_t length;        /* 当前有效字节数 */
  uint32_t timestamp_ms;  /* 接收或发送时刻，单位 ms */
} Bsp_RawFrame_t;
```

BSP 层只返回原始字节、传输状态和超时状态，不直接返回 GNSS 坐标、气象数值、Remote ID 状态等业务数据。

### 2.2 传感器采样结构

当前定义位置：

```text
Sensor_Driver/Inc/sensor_data.h
```

当前已实现：

```c
#define SENSOR_UNIT_LEN 8U

typedef enum {
  SENSOR_TYPE_UNKNOWN = 0,  /* 未分类设备 */
  SENSOR_TYPE_GNSS,         /* 导航定位模块 */
  SENSOR_TYPE_REMOTE_ID,    /* Remote ID 或监视类模块 */
  SENSOR_TYPE_WEATHER,      /* 气象或环境传感器 */
  SENSOR_TYPE_POWER,        /* 电源、电压、电流检测 */
  SENSOR_TYPE_RELAY_IO      /* 继电器、开关量、普通 IO */
} Sensor_Type_t;

typedef enum {
  SENSOR_VALUE_NONE = 0,    /* 无有效数据 */
  SENSOR_VALUE_FLOAT,       /* 单精度浮点值 */
  SENSOR_VALUE_INT32,       /* 有符号整型值 */
  SENSOR_VALUE_UINT32,      /* 无符号整型值 */
  SENSOR_VALUE_BOOL,        /* 布尔或开关量 */
  SENSOR_VALUE_GEO          /* 经纬高数据 */
} Sensor_ValueType_t;

typedef struct
{
  int32_t latitude_deg_e7;   /* 纬度，单位 degree * 1e7 */
  int32_t longitude_deg_e7;  /* 经度，单位 degree * 1e7 */
  int32_t altitude_mm;       /* 高度，单位 mm */
} Sensor_GeoValue_t;

typedef union {
  float f32;                 /* 浮点采样值 */
  int32_t i32;               /* 有符号整型采样值 */
  uint32_t u32;              /* 无符号整型采样值 */
  uint8_t boolean;           /* 开关量 */
  Sensor_GeoValue_t geo;     /* 经纬高组合值 */
} Sensor_Value_t;

typedef struct
{
  uint16_t device_id;              /* 设备编号，由项目接口表统一分配 */
  Sensor_Type_t sensor_type;       /* 传感器类型 */
  uint16_t channel;                /* 通道号，例如电源第 1 路、电机第 2 路 */
  Sensor_ValueType_t value_type;   /* value 联合体中当前有效的数据类型 */
  uint32_t timestamp_ms;           /* 采样时间戳，单位 ms */
  uint32_t status;                 /* 设备自定义状态位，0 表示正常 */
  char unit[SENSOR_UNIT_LEN];      /* 单位，例如 "V"、"m/s"、"deg" */
  Sensor_Value_t value;            /* 采样值 */
} Sensor_Sample_t;
```

使用约束：

- 一个传感器可以一次返回多个 `Sensor_Sample_t`，例如气象模块同时返回温度、湿度、气压。
- `device_id` 必须来自统一接口表，不能由驱动随意硬编码冲突编号。
- `status` 用于传感器采样状态，不替代系统模块状态 `App_ModuleStatus_t`。
- 经纬高使用整数定点表示，避免在协议传输中依赖 double。

### 3.4 正式 CNS 通信帧

建议定义位置：

```text
App/Inc/cns_frame.h
```

建议结构：

```c
#define CNS_FRAME_HEADER_0 0xAAU
#define CNS_FRAME_HEADER_1 0x55U
#define CNS_FRAME_VERSION  0x01U
#define CNS_FRAME_PAYLOAD_MAX_LEN 128U

typedef struct
{
  uint8_t header0;                         /* 固定 0xAA */
  uint8_t header1;                         /* 固定 0x55 */
  uint8_t version;                         /* 协议版本 */
  CNS_MessageType_t msg_type;              /* 消息类型 */
  uint16_t sequence;                       /* 帧序号，用于去重和应答匹配 */
  uint8_t source;                          /* 源节点，取值见 CNS_NodeType_t */
  uint8_t target;                          /* 目标节点，取值见 CNS_NodeType_t */
  uint32_t timestamp_ms;                   /* 主控本地时间戳 */
  uint16_t payload_len;                    /* payload 有效长度 */
  uint8_t payload[CNS_FRAME_PAYLOAD_MAX_LEN]; /* 负载数据 */
  uint16_t checksum;                       /* 校验值，建议 CRC16 */
} CNS_Frame_t;
```

建议编解码接口：

```c
typedef enum {
  CNS_FRAME_OK = 0,
  CNS_FRAME_INVALID_PARAM,
  CNS_FRAME_INVALID_LENGTH,
  CNS_FRAME_INVALID_HEADER,
  CNS_FRAME_INVALID_VERSION,
  CNS_FRAME_INVALID_CHECKSUM
} CNS_FrameResult_t;

CNS_FrameResult_t CNS_EncodeFrame(const CNS_Frame_t *frame,
                                  uint8_t *buffer,
                                  uint16_t buffer_size,
                                  uint16_t *out_len);

CNS_FrameResult_t CNS_DecodeFrame(const uint8_t *buffer,
                                  uint16_t length,
                                  CNS_Frame_t *frame);
```

### 3.5 主控对网关的负载来源

主控和网关之间统一发送 `CNS_Frame_t`，但负载来源可以不同：

| 负载来源 | 当前定义位置 | 对外消息类型 |
| --- | --- | --- |
| 传感器采样 | `Sensor_Driver/Inc/sensor_data.h` 的 `Sensor_Sample_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_SENSOR_DATA` |
| 系统模块状态 | `App/Inc/app_status.h` 的 `App_ModuleStatus_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_STATUS` |
| 告警状态 | `App/Inc/app_alarm.h` 的 `App_AlarmRecord_t`，对外错误码使用 `App/Inc/app_error.h` 的 `App_ErrorCode_t` | App 层汇总到 `CNS_State`，必要时封装为 `CNS_MSG_ALARM` |
| 心跳 | `App/Protocol` 后续定义 | `CNS_MSG_HEARTBEAT` |
| 参数配置 | 后续 `App/Inc/app_config.h` | `CNS_MSG_CONFIG` |
| 控制命令 | 后续 `App/Inc/cns_message.h` | `CNS_MSG_CONTROL` |

`App_ModuleStatus_t` 只表达系统健康状态，不承载传感器采样值。

### 4.2 应用层汇总状态

`CNS_State` 是应用层根据传感器采样、系统模块状态、告警状态、LoRa 链路统计和参数状态生成的汇总快照。各模块只更新自身状态或通过统一接口输出数据，不直接写入 `CNS_State`。

日志、HMI 刷新和对外协议上报应读取同一份 `CNS_State` 快照。多任务访问时应采用互斥锁、临界区、消息队列或双缓冲快照，避免日志和上报读到不一致的数据组合。

### 4.3 告警项和错误码

当前定义位置：

```text
App/Inc/app_alarm.h
App/Inc/app_error.h
```

`App_AlarmId_t` 是系统内部告警项，用于 active mask、告警清除和告警记录索引；`App_ErrorCode_t` 是对外稳定错误码，用于 HMI、CSV 日志、LoRa/网关上报和人工排查；`payload` 保存模块差异化异常上下文；`detail` 是从 payload 提取出的简化诊断值，不作为主错误码。

`APP_ERROR_OK` 只表示无异常，不能作为告警触发时的错误码占位值。已有具体标准错误码时调用 `App_AlarmRaise()`；只有告警项、没有具体错误码时调用 `App_AlarmRaiseDefault()`，由告警模块内部默认映射表生成对外错误码。

告警运行时统一采用队列入口：

```text
模块 -> 构造 App_AlarmMsg_t -> App_AlarmPost() -> 告警队列 -> App_AlarmTask() -> App_AlarmRecord_t
```

`App_AlarmRaise()`、`App_AlarmRaiseDefault()` 和 `App_AlarmClear()` 只是便捷封装，内部仍然构造 `App_AlarmMsg_t` 后投递队列。模块需要携带传感器、通信、存储或系统类上下文时，应使用 `App_AlarmBuildRaiseMsg()` 或 `App_AlarmBuildRaiseDefaultMsg()` 构造消息，再调用 `App_AlarmPost()`。


```c
typedef enum {
  COMM_LINK_UART_GATEWAY = 0,  /* 连接 Linux 网关的 UART */

  COMM_LINK_CAN_GATEWAY,       /* CAN 网关或 CAN 设备 */

} Comm_LinkType_t;

typedef enum {
  COMM_LINK_OK = 0,
  COMM_LINK_WARNING,
  COMM_LINK_NOT_READY,
  COMM_LINK_ERROR
} Comm_LinkResult_t;

typedef struct
{
  uint8_t link_id;       /* 链路编号 */
  Comm_LinkType_t type;  /* 链路类型 */
  const char *name;      /* 链路名称 */
  Comm_LinkResult_t (*init)(void);
  Comm_LinkResult_t (*self_check)(uint32_t *error_code);
```

`Comm_Driver` 负责选择链路和收发字节；`App/Protocol` 负责 `CNS_Frame_t` 编解码。两者不要混在一个模块里。

## 6. 定义位置总览

| 数据结构 | 当前/建议位置 | 作用 |
| --- | --- | --- |
| `Bsp_BusType_t` | 建议 `Bsp/Inc/bsp_bus.h` | 底层总线类型 |
| `Bsp_RawFrame_t` | 建议 `Bsp/Inc/bsp_bus.h` | 原始收发帧 |
| `Sensor_Type_t` | `Sensor_Driver/Inc/sensor_data.h` | 传感器类型 |
| `Sensor_Sample_t` | `Sensor_Driver/Inc/sensor_data.h` | 统一传感器采样 |
| `Sensor_Driver_t` | `Sensor_Driver/Inc/sensor_driver.h` | 统一传感器驱动接口 |
| `App_ModuleStatus_t` | `App/Inc/app_status.h` | 系统模块状态 |
| `App_AlarmId_t` | `App/Inc/app_alarm.h` | 系统内部告警项 |
| `App_ErrorCode_t` | `App/Inc/app_error.h` | HMI、CSV、上报使用的标准错误码 |
| `App_AlarmRecord_t` | `App/Inc/app_alarm.h` | 告警运行时记录 |
| `App_ProtocolMessage_t` | `App/Inc/app_protocol.h` | 当前 demo 协议消息 |

| `Comm_LinkDriver_t` | 建议 `Comm_Driver/Inc/comm_link.h` | 网关/无线链路驱动接口 |

## 7. 迁移和扩展约束

- 具体传感器驱动不得直接调用显示、日志、协议发送接口。
- 传感器采样数据统一通过 `Sensor_Sample_t` 输出。
- 系统健康状态统一通过 `App_ModuleStatus_t` 输出。
- 主控对外通信统一通过 `CNS_Frame_t` 输出。
- HAL 相关句柄和寄存器操作应限制在 `Bsp` 内部。
- 网关、LoRa、Mesh 等链路应放入后续 `Comm_Driver`，不应写入 `Sensor_Driver`。
