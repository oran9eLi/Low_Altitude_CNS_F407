# 研发协作规则

## 基本原则

1. 需求先冻结，再并行开发。
2. 接口表先冻结，再接线和写驱动。
3. 单模块先验证，再进入主工程。
4. 文档和代码同步提交。
5. 问题必须闭环，禁止口头说明。

## 开发流程

```text
拉取 develop
-> 新建 feature 分支
-> 单模块开发
-> 单模块测试
-> 更新文档
-> 提交合并请求
-> 代码评审
-> 合入 develop
-> 系统联调
-> 阶段发布
```

## 提交前缀规则

提交前缀采用 Conventional Commits 风格，格式固定为：

```text
<type>: <中文提交说明>
```

允许使用的 `type` 如下：

| 前缀 | 适用场景 |
| --- | --- |
| feat | 新增功能、模块或对外能力 |
| fix | 修复缺陷、异常行为或编译问题 |
| docs | 修改说明文档、协议表、接口表或协作规则 |
| chore | 仓库初始化、忽略规则、工具配置、非功能性维护 |
| refactor | 重构代码结构，不改变外部行为 |
| test | 新增或修改测试、验证脚本、测试记录 |
| build | 修改构建配置、Keil 工程配置、链接脚本或依赖组织 |
| ci | 修改自动化检查、流水线或发布脚本 |
| style | 仅调整格式、命名、注释排版，不改变逻辑 |
| perf | 性能、资源占用或实时性优化 |
| revert | 回退已有提交 |

## 提交信息规则

1. 提交信息必须使用中文说明，前缀保留英文小写。
2. 冒号后必须保留一个空格，例如 `chore: 初始化仓库`。
3. 单行提交说明应概括本次变更结果，不写过程描述。
4. 每次提交应保持职责单一，避免把无关文档、代码和构建产物混在一起。
5. 涉及固件代码、Keil 工程配置或接口协议变化时，应在提交前完成对应验证，并同步更新文档。
6. 禁止使用含糊提交信息，例如 `update`、`修改`、`临时提交`、`wip`。

示例：

```text
chore: 初始化仓库
docs: 更新协作规则
feat: 增加 LoRa 链路状态上报
fix: 修复传感器状态汇总空指针检查
build: 补充 Keil 工程源文件列表
```

## 模块进入主工程的条件

1. 接线定义明确。
2. 初始化函数可用。
3. Demo 已单独跑通。
4. 有测试记录。
5. 提供标准 API。
6. 错误状态进入错误码表。
