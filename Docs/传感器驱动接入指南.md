# 传感器驱动接入指南

本文面向具体传感器驱动开发人员，说明新增传感器驱动时需要实现的接口、状态填写规则、采样数据格式和注册表接入方式。

驱动开发人员只需要遵守本文接口约定，不直接修改系统任务、日志、显示、告警或对外通信协议。

## 1. 分层边界

```text
App
  -> Sensor_RegistryReadAll() / Sensor_RegistrySelfCheckAbnormal()
      -> Sensor_Driver_t.read() / Sensor_Driver_t.self_check()
          -> 具体传感器驱动
              -> Bsp UART/CAN/I2C/SPI 等底层接口
```

职责边界：

- `Bsp`：只负责底层收发和板级接口，不解析传感器业务含义。
- `Sensor_Driver`：负责具体传感器初始化、自检、协议解析、数据转换和统一采样输出。
- `App`：负责任务调度、系统状态、异常告警、日志、显示和对外协议。

驱动中不要直接调用显示、日志、协议发送、`App_StatusSet()`、`App_AlarmBuildMsg()` 或 `App_AlarmPost()`。

## 2. 文件放置

新增传感器驱动建议按以下方式命名：

```text
Sensor_Driver/Inc/xxx_driver.h
Sensor_Driver/Src/xxx_driver.c
```

驱动文件加入后，需要把对应 `.c` 文件加入 Keil 工程的 `Sensor_Driver` 分组。

## 3. 统一状态结构

当前状态结构定义在 `Sensor_Driver/Inc/sensor_driver.h`：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Type_t sensor_type;
  Sensor_Severity_t severity;
  App_ErrorCode_t code;
  uint32_t driver_error;
} Sensor_Status_t;
```

字段规则：

| 字段 | 规则 |
| --- | --- |
| `device_id` | 必须与驱动句柄 `Sensor_Driver_t.device_id` 一致 |
| `sensor_type` | 必须与驱动句柄 `Sensor_Driver_t.sensor_type` 一致 |
| `severity` | 表达影响级别，不表达具体故障类别 |
| `code` | 正常时填 `ERR_OK`；异常时只填 `ERR_SENSOR_*` 范围内的统一错误码 |
| `driver_error` | 可选调试细节；没有底层细节时填 `0U` |

`driver_error` 不是必须字段。驱动能拿到底层细节时，可以填 HAL 返回值、`HAL_xxx_GetError()`、芯片状态寄存器、协议解析错误或驱动自定义阶段码；如果没有细节，填 `0U` 即可。上层显示、日志和上报只用 `code` 和 `severity` 判断主错误。

## 4. 严重程度

```text
SENSOR_SEVERITY_NORMAL
  正常。

SENSOR_SEVERITY_GENERAL
  可继续运行，但数据质量下降或存在轻微异常。

SENSOR_SEVERITY_IMPORTANT
  影响实验或数据可信度，需要提示或进入降级策略。

SENSOR_SEVERITY_CRITICAL
  影响安全或主系统运行，需要停机或人工处理。
```

驱动返回值应稳定，不要因为单次无新数据就频繁返回 `SENSOR_SEVERITY_CRITICAL`。

## 5. 传感器错误码

传感器驱动直接使用统一错误码，不再返回独立的 `Sensor_FaultReason_t`。

| 错误码 | 含义 | 底层细节放置位置 |
| --- | --- | --- |
| `ERR_SENSOR_OFFLINE` | 传感器离线或长时间无响应 | `driver_error` |
| `ERR_SENSOR_NO_DATA` | 当前没有可用数据 | `driver_error` |
| `ERR_SENSOR_DATA_INVALID` | 数据校验失败、格式错误或数值不可信 | `driver_error` |
| `ERR_SENSOR_NO_FIX` | GNSS 无有效定位 | `driver_error` |
| `ERR_SENSOR_INIT_FAILED` | 初始化失败 | `driver_error` |
| `ERR_SENSOR_TIMEOUT` | 采样或通信超时 | `driver_error` |

不要把 I2C、SPI、UART 等总线类型拆成对外错误码。总线 NACK、超时寄存器、芯片状态等信息放入 `driver_error`。

## 6. 必须实现的函数签名

### 6.1 初始化

```c
Sensor_Severity_t Xxx_Init(Sensor_Status_t *status);
```

初始化成功时填写：

```c
status->severity = SENSOR_SEVERITY_NORMAL;
status->code = ERR_OK;
status->driver_error = 0U;
```

初始化失败时按实际情况填写 `ERR_SENSOR_INIT_FAILED`、`ERR_SENSOR_OFFLINE` 或 `ERR_SENSOR_TIMEOUT`。

### 6.2 自检

```c
Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status);
```

自检负责输出当前设备健康状态。示例：

```c
Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status)
{
  if (status != NULL)
  {
    status->device_id = SENSOR_DEVICE_XXX_1;
    status->sensor_type = SENSOR_TYPE_UNKNOWN;
    status->severity = SENSOR_SEVERITY_NORMAL;
    status->code = ERR_OK;
    status->driver_error = 0U;
  }

  if (device_timeout)
  {
    if (status != NULL)
    {
      status->severity = SENSOR_SEVERITY_IMPORTANT;
      status->code = ERR_SENSOR_TIMEOUT;
      status->driver_error = 1U;
    }
    return SENSOR_SEVERITY_IMPORTANT;
  }

  return SENSOR_SEVERITY_NORMAL;
}
```

### 6.3 读取

```c
Sensor_Severity_t Xxx_Read(Sensor_Sample_t *samples,
                           uint16_t max_count,
                           uint16_t *out_count,
                           Sensor_Status_t *status);
```

基本要求：

- 如果 `out_count != NULL`，函数开始时先写 `*out_count = 0U`。
- 不允许写超过 `max_count` 的采样数量。
- `samples == NULL` 或 `max_count == 0U` 时返回异常，并把 `status->code` 设置为 `ERR_SENSOR_NO_DATA` 或 `ERR_SENSOR_DATA_INVALID`。
- 没有新数据时按设备语义返回 `SENSOR_SEVERITY_GENERAL` 或 `SENSOR_SEVERITY_IMPORTANT`，并填写 `ERR_SENSOR_NO_DATA`。

### 6.4 状态查询

```c
Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status);
```

`get_status` 应尽量只返回当前缓存状态，不主动触发耗时总线读写。如果当前驱动没有缓存状态，可以临时复用 `self_check`，后续再细化。

## 7. 驱动句柄

每个驱动必须提供一个 `Sensor_Driver_t`：

```c
const Sensor_Driver_t g_gnss_driver =
{
  .device_id = SENSOR_DEVICE_GNSS_1,
  .sensor_type = SENSOR_TYPE_GNSS,
  .name = "gnss",
  .init = Gnss_Init,
  .self_check = Gnss_SelfCheck,
  .read = Gnss_Read,
  .get_status = Gnss_GetStatus
};
```

`name` 会进入传感器异常 payload，供显示、日志和上报使用。

## 8. 采样数据

采样结构定义在 `Sensor_Driver/Inc/sensor_data.h`：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Type_t sensor_type;
  uint16_t channel;
  Sensor_ValueType_t value_type;
  uint32_t timestamp_ms;
  uint32_t status;
  char unit[SENSOR_UNIT_LEN];
  Sensor_Value_t value;
} Sensor_Sample_t;
```

约定：

- `device_id` 和 `sensor_type` 必须与驱动句柄一致。
- 一个驱动可以一次输出多条采样记录，例如气象模块同时返回温度、湿度、气压。
- `status` 是采样状态位，不替代 `Sensor_Status_t.code`。
- 经纬高使用整数定点表示，避免协议传输依赖 double。

## 9. 设备 ID 分配

`device_id` 由项目统一分配。具体传感器驱动只引用已分配的 ID，不自行新增裸数字。

当前定义位置：

```text
Sensor_Driver/Inc/sensor_device_id.h
```

## 10. 接入注册表

驱动实现完成后，在 `Sensor_Driver/Src/sensor_registry.c` 包含驱动头文件，并把驱动句柄挂入 `s_sensor_drivers[]`：

```c
static const Sensor_Driver_t *const s_sensor_drivers[] =
{
  &g_gnss_driver,
  &g_bme280_driver,
  &g_power_driver
};
```

系统会通过注册表统一调用：

```text
Sensor_RegistryInitAll()
Sensor_RegistrySelfCheckAll()
Sensor_RegistrySelfCheckAbnormal()
Sensor_RegistryReadAll()
```

`Sensor_RegistrySelfCheckAbnormal()` 只输出异常传感器。应用注册表会把每个异常传感器构造成一条 `App_AlarmMsg_t`，并用 `source + code + instance_id` 维护活动异常；当传感器恢复时，应用注册表会投递 CLEAR 消息。

## 11. 禁止事项

- 不要直接创建 FreeRTOS 任务。
- 不要直接写日志文件。
- 不要直接刷新 DWIN 显示。
- 不要直接拼接主控到网关的协议帧。
- 不要直接调用 `App_StatusSet()`、`App_AlarmBuildMsg()` 或 `App_AlarmPost()`。
- 不要在驱动接口中暴露 HAL 句柄。
- 不要在驱动里使用未统一分配的 `device_id` 裸数字。

## 12. 最小模板

```c
#include "xxx_driver.h"

#include <string.h>
#include "sensor_device_id.h"

static App_ErrorCode_t s_code = ERR_OK;
static uint32_t s_driver_error = 0U;

static void Xxx_FillStatus(Sensor_Status_t *status, Sensor_Severity_t severity)
{
  if (status == NULL)
  {
    return;
  }

  status->device_id = SENSOR_DEVICE_XXX_1;
  status->sensor_type = SENSOR_TYPE_UNKNOWN;
  status->severity = severity;
  status->code = (severity == SENSOR_SEVERITY_NORMAL) ? ERR_OK : s_code;
  status->driver_error = s_driver_error;
}

Sensor_Severity_t Xxx_Init(Sensor_Status_t *status)
{
  s_code = ERR_OK;
  s_driver_error = 0U;
  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status)
{
  if (s_driver_error != 0U)
  {
    s_code = ERR_SENSOR_DATA_INVALID;
    Xxx_FillStatus(status, SENSOR_SEVERITY_GENERAL);
    return SENSOR_SEVERITY_GENERAL;
  }

  s_code = ERR_OK;
  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_Read(Sensor_Sample_t *samples,
                           uint16_t max_count,
                           uint16_t *out_count,
                           Sensor_Status_t *status)
{
  if (out_count != NULL)
  {
    *out_count = 0U;
  }

  if ((samples == NULL) || (max_count == 0U))
  {
    s_code = ERR_SENSOR_NO_DATA;
    s_driver_error = 1U;
    Xxx_FillStatus(status, SENSOR_SEVERITY_IMPORTANT);
    return SENSOR_SEVERITY_IMPORTANT;
  }

  samples[0].device_id = SENSOR_DEVICE_XXX_1;
  samples[0].sensor_type = SENSOR_TYPE_UNKNOWN;
  samples[0].channel = 0U;
  samples[0].value_type = SENSOR_VALUE_FLOAT;
  samples[0].timestamp_ms = 0U;
  samples[0].status = 0U;
  strncpy(samples[0].unit, "-", SENSOR_UNIT_LEN);
  samples[0].value.f32 = 0.0f;

  if (out_count != NULL)
  {
    *out_count = 1U;
  }

  s_code = ERR_OK;
  s_driver_error = 0U;
  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status)
{
  return Xxx_SelfCheck(status);
}
```
