# 传感器驱动接入指南

本文面向具体传感器驱动开发人员，说明新增传感器驱动时必须遵守的接口、状态缓存、采样输出和注册方式。

本文以目标框架接口为准，驱动层重构后应直接按本文实现。

驱动开发人员只负责具体设备驱动，不直接创建系统任务，不直接写日志、显示、告警或对外通信协议。

## 1. 分层边界

```text
App / SensorService / 上层采样调度者
  -> Sensor_Registry
      -> Sensor_Driver_t
          -> 具体传感器驱动
              -> BSP / Bus / HAL
```

职责边界：

| 层级 | 职责 |
|---|---|
| `BSP/Bus` | 提供 UART、I2C、SPI、CAN 等底层收发能力，不解析传感器业务含义 |
| `Sensor_Driver` | 初始化具体设备，解析原始数据，输出统一采样和缓存状态 |
| `Sensor_Registry` | 统一管理驱动句柄，批量初始化、启动自检、采样和状态读取 |
| `App` | 负责任务调度、系统状态、异常告警、日志、显示和对外协议 |

驱动中不要直接调用 `App_StatusSet()`、`App_AlarmPost()`、显示刷新、日志写入或协议发送接口。

## 2. 文件放置

新增传感器驱动建议按以下方式命名：

```text
Sensor_Driver/Inc/xxx_driver.h
Sensor_Driver/Src/xxx_driver.c
```

驱动文件加入后，需要把对应 `.c` 文件加入 Keil 工程的 `Sensor_Driver` 分组。

## 3. 设备编号和名称

每个具体设备实例必须有唯一 `device_id`，定义位置：

```text
Sensor_Driver/Inc/sensor_device_id.h
```

规则：

- 驱动只引用已分配的 `SENSOR_DEVICE_xxx`，不要在驱动里写裸数字。
- `device_id` 用于告警实例、采样来源、协议上报和日志定位。
- `name` 用于显示、日志和告警 payload，建议使用稳定 ASCII 名称，例如 `"GNSS1"`、`"BME280_1"`。
- V1.0 运行期接口不保留 `sensor_type`，设备类别后续如有需要由独立设备信息表维护。

## 4. 统一错误码

传感器驱动直接使用系统统一错误码，不再定义独立的驱动层故障枚举。

| 错误码 | 含义 | 使用建议 |
|---|---|---|
| `ERR_OK` | 正常 | 当前设备无主错误 |
| `ERR_SENSOR_OFFLINE` | 传感器离线或长时间无响应 | 设备不存在、断线、连续无响应 |
| `ERR_SENSOR_NO_DATA` | 当前没有可用数据 | 没有新数据或缓冲区为空 |
| `ERR_SENSOR_DATA_INVALID` | 数据无效 | 校验失败、格式错误、数值越界 |
| `ERR_SENSOR_NO_FIX` | GNSS 无有效定位 | GNSS 类设备专用 |
| `ERR_SENSOR_INIT_FAILED` | 初始化失败 | 复位、配置或启动失败 |
| `ERR_SENSOR_TIMEOUT` | 采样或通信超时 | 单次访问超时 |

不要把 I2C、SPI、UART 等总线类型拆成对外错误码。HAL 错误、芯片状态寄存器、协议解析阶段码等放入 `driver_error`。

## 5. 严重程度

`Sensor_Severity_t` 只表达影响级别，不表达具体故障原因。

| 枚举 | 含义 |
|---|---|
| `SENSOR_SEVERITY_NORMAL` | 正常 |
| `SENSOR_SEVERITY_GENERAL` | 一般异常，可继续运行 |
| `SENSOR_SEVERITY_IMPORTANT` | 重要异常，影响实验或数据可信度 |
| `SENSOR_SEVERITY_CRITICAL` | 严重异常，需要停机或人工处理 |

驱动返回值应稳定，不要因为单次无新数据就频繁返回严重异常。具体故障原因以 `Sensor_Status_t.code` 为准。

## 6. 状态结构

目标状态结构：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Severity_t severity;
  App_ErrorCode_t code;
  uint32_t driver_error;
} Sensor_Status_t;
```

字段规则：

| 字段 | 规则 |
|---|---|
| `device_id` | 必须与驱动句柄 `Sensor_Driver_t.device_id` 一致 |
| `severity` | 当前主错误的影响级别，正常时为 `SENSOR_SEVERITY_NORMAL` |
| `code` | 正常时填 `ERR_OK`，异常时填 `ERR_SENSOR_*` |
| `driver_error` | 可选驱动细节；没有底层细节时填 `0U` |

`driver_error` 不是必填字段。驱动能拿到底层细节时，可以填 HAL 返回值、`HAL_xxx_GetError()`、芯片状态寄存器、协议解析错误或驱动自定义阶段码；如果没有细节，填 `0U` 即可。上层只用 `code` 和 `severity` 判断主错误。

## 7. 采样结构

目标采样结构：

```c
typedef struct
{
  uint16_t device_id;
  uint16_t channel;
  Sensor_ValueType_t value_type;
  uint32_t timestamp_ms;
  uint8_t status;
  char unit[SENSOR_UNIT_LEN];
  Sensor_Value_t value;
} Sensor_Sample_t;
```

字段规则：

| 字段 | 规则 |
|---|---|
| `device_id` | 设备实例编号 |
| `channel` | 同一设备的采样通道，从 `0` 开始分配 |
| `value_type` | 指明 `value` 联合体中哪个字段有效 |
| `timestamp_ms` | 本次采样时间，单位 ms |
| `status` | `0=正常, 1=异常`，只表达本条样本是否可用 |
| `unit` | 单位字符串，长度不超过 `SENSOR_UNIT_LEN - 1` |
| `value` | 采样值 |

多传感器和多通道通过 `device_id + channel + value_type` 区分。一个驱动一次可以输出多条 `Sensor_Sample_t`：

```text
BME280_1:
  device_id=SENSOR_DEVICE_BME280_1, channel=0, value_type=FLOAT, unit="degC"
  device_id=SENSOR_DEVICE_BME280_1, channel=1, value_type=FLOAT, unit="%RH"
  device_id=SENSOR_DEVICE_BME280_1, channel=2, value_type=FLOAT, unit="hPa"

GNSS_1:
  device_id=SENSOR_DEVICE_GNSS_1, channel=0, value_type=GEO, unit="geo"
```

`status` 不替代 `Sensor_Status_t.code`。当任一样本 `status == 1` 时，上层会读取该驱动缓存状态，获取具体 `code`、`severity` 和 `driver_error`。

## 8. 驱动接口

目标驱动接口：

```c
typedef Sensor_Severity_t (*Sensor_InitFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_SelfCheckFn)(Sensor_Status_t *status);
typedef Sensor_Severity_t (*Sensor_ReadFn)(Sensor_Sample_t *samples, uint16_t max_count, uint16_t *out_count);
typedef Sensor_Severity_t (*Sensor_GetStatusFn)(Sensor_Status_t *status);

typedef struct
{
  uint16_t device_id;
  const char *name;
  Sensor_InitFn init;
  Sensor_SelfCheckFn self_check;
  Sensor_ReadFn read;
  Sensor_GetStatusFn get_status;
} Sensor_Driver_t;
```

函数职责：

| 函数 | 是否可访问硬件 | 调用时机 | 职责 |
|---|---|---|---|
| `init` | 可以 | 系统初始化 | 初始化设备、内部变量和状态缓存 |
| `self_check` | 可以 | 上电自检阶段 | 在 `init` 之后确认设备可访问，能读到基本数据，刷新状态缓存 |
| `read` | 可以 | 运行期采样 | 读取并解析数据，输出样本，刷新状态缓存 |
| `get_status` | 不可以 | 状态汇总或异常查询 | 只返回最近一次缓存状态 |

`self_check` 沿用既有命名，但语义收敛为启动自检：只在 `init` 之后、进入运行期之前调用，不用于运行期周期自检。运行期异常由 `read` 发现并刷新状态缓存。

## 9. 状态缓存要求

每个驱动内部必须维护最近一次状态，例如：

```c
static Sensor_Status_t s_status;
```

要求：

- `init`、`self_check`、`read` 都可以刷新 `s_status`。
- `get_status` 只复制 `s_status`，不得访问总线。
- `read` 发现异常时，必须先刷新 `s_status`，再返回。
- 正常恢复时，`read` 也必须把 `s_status.code` 刷回 `ERR_OK`。

推荐辅助函数：

```c
static void Xxx_SetStatus(Sensor_Severity_t severity, App_ErrorCode_t code, uint32_t driver_error)
{
  s_status.device_id = SENSOR_DEVICE_XXX_1;
  s_status.severity = severity;
  s_status.code = code;
  s_status.driver_error = driver_error;
}

static Sensor_Severity_t Xxx_CopyStatus(Sensor_Status_t *status)
{
  if (status != NULL) {
    *status = s_status;
  }
  return s_status.severity;
}
```

## 10. 采样读取规则

`read` 的基本规则：

- 函数开始时，如果 `out_count != NULL`，先写 `*out_count = 0U`。
- 不允许写超过 `max_count` 的采样数量。
- `samples == NULL` 或 `max_count == 0U` 时返回异常，并刷新缓存状态。
- 有多个采样项时按数组连续写入。
- 多通道传感器必须先判断本次完整输出需要的样本数量。如果 `max_count` 不足以容纳完整一组采样，不输出半包，保持 `out_count=0U`，刷新缓存状态并返回异常。
- 某个通道异常时，可以只输出该通道异常样本，也可以输出已成功读取的其他通道正常样本。
- 如果整机没有读到任何有效数据，`out_count` 保持 `0U`，状态缓存填写具体错误码。

调用方只保证给出一个样本缓冲区，驱动必须根据 `max_count` 控制写入数量。上层后续会把采样快照交给 Protocol 任务构造 CNS `SENSOR` 帧，驱动不参与协议组帧。

多通道读取示例：

```c
#define BME280_SAMPLE_COUNT 3U

static Sensor_Severity_t Bme280_Read(Sensor_Sample_t *samples, uint16_t max_count, uint16_t *out_count)
{
  if (out_count != NULL) {
    *out_count = 0U;
  }

  if ((samples == NULL) || (max_count < BME280_SAMPLE_COUNT)) {
    Bme280_SetStatus(SENSOR_SEVERITY_IMPORTANT, ERR_SENSOR_NO_DATA, 0U);
    return SENSOR_SEVERITY_IMPORTANT;
  }

  /* 一次访问硬件，读取温度、湿度、气压。 */

  samples[0].device_id = SENSOR_DEVICE_BME280_1;
  samples[0].channel = 0U;
  samples[0].value_type = SENSOR_VALUE_FLOAT;
  samples[0].timestamp_ms = timestamp_ms;
  samples[0].status = 0U;
  strncpy(samples[0].unit, "degC", SENSOR_UNIT_LEN);
  samples[0].unit[SENSOR_UNIT_LEN - 1U] = '\0';
  samples[0].value.f32 = temp;

  samples[1].device_id = SENSOR_DEVICE_BME280_1;
  samples[1].channel = 1U;
  samples[1].value_type = SENSOR_VALUE_FLOAT;
  samples[1].timestamp_ms = timestamp_ms;
  samples[1].status = 0U;
  strncpy(samples[1].unit, "%RH", SENSOR_UNIT_LEN);
  samples[1].unit[SENSOR_UNIT_LEN - 1U] = '\0';
  samples[1].value.f32 = humi;

  samples[2].device_id = SENSOR_DEVICE_BME280_1;
  samples[2].channel = 2U;
  samples[2].value_type = SENSOR_VALUE_FLOAT;
  samples[2].timestamp_ms = timestamp_ms;
  samples[2].status = 0U;
  strncpy(samples[2].unit, "hPa", SENSOR_UNIT_LEN);
  samples[2].unit[SENSOR_UNIT_LEN - 1U] = '\0';
  samples[2].value.f32 = pressure;

  if (out_count != NULL) {
    *out_count = BME280_SAMPLE_COUNT;
  }

  Bme280_SetStatus(SENSOR_SEVERITY_NORMAL, ERR_OK, 0U);
  return SENSOR_SEVERITY_NORMAL;
}
```

## 11. 接入注册表

每个驱动必须导出一个全局只读句柄：

```c
const Sensor_Driver_t g_xxx_driver =
{
  .device_id = SENSOR_DEVICE_XXX_1,
  .name = "XXX_1",
  .init = Xxx_Init,
  .self_check = Xxx_SelfCheck,
  .read = Xxx_Read,
  .get_status = Xxx_GetStatus
};
```

驱动实现完成后，由框架维护者在注册表中挂入：

```c
static const Sensor_Driver_t *const s_sensor_drivers[] =
{
  &g_gnss_driver,
  &g_bme280_driver,
  &g_power_driver
};
```

具体驱动同事不需要修改任务调度、告警模块或协议模块。

## 12. 禁止事项

- 不要直接创建 FreeRTOS 任务。
- 不要直接写日志文件。
- 不要直接刷新 DWIN 显示。
- 不要直接拼接 CNS 协议帧。
- 不要直接调用 `App_StatusSet()`、`App_AlarmBuildMsg()` 或 `App_AlarmPost()`。
- 不要在驱动接口中暴露 HAL 句柄。
- 不要在 `get_status` 中访问硬件。
- 不要在驱动里使用未统一分配的 `device_id` 裸数字。
- 不要新增驱动私有错误码体系替代 `ERR_SENSOR_*`。

## 13. 最小模板

```c
#include "xxx_driver.h"

#include <string.h>
#include "sensor_device_id.h"

static Sensor_Status_t s_status;

static void Xxx_SetStatus(Sensor_Severity_t severity, App_ErrorCode_t code, uint32_t driver_error)
{
  s_status.device_id = SENSOR_DEVICE_XXX_1;
  s_status.severity = severity;
  s_status.code = code;
  s_status.driver_error = driver_error;
}

static Sensor_Severity_t Xxx_CopyStatus(Sensor_Status_t *status)
{
  if (status != NULL) {
    *status = s_status;
  }
  return s_status.severity;
}

static Sensor_Severity_t Xxx_Init(Sensor_Status_t *status)
{
  Xxx_SetStatus(SENSOR_SEVERITY_NORMAL, ERR_OK, 0U);
  return Xxx_CopyStatus(status);
}

static Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status)
{
  if (device_not_found) {
    Xxx_SetStatus(SENSOR_SEVERITY_IMPORTANT, ERR_SENSOR_OFFLINE, 0U);
    return Xxx_CopyStatus(status);
  }

  Xxx_SetStatus(SENSOR_SEVERITY_NORMAL, ERR_OK, 0U);
  return Xxx_CopyStatus(status);
}

static Sensor_Severity_t Xxx_Read(Sensor_Sample_t *samples, uint16_t max_count, uint16_t *out_count)
{
  if (out_count != NULL) {
    *out_count = 0U;
  }

  if ((samples == NULL) || (max_count == 0U)) {
    Xxx_SetStatus(SENSOR_SEVERITY_IMPORTANT, ERR_SENSOR_NO_DATA, 0U);
    return s_status.severity;
  }

  samples[0].device_id = SENSOR_DEVICE_XXX_1;
  samples[0].channel = 0U;
  samples[0].value_type = SENSOR_VALUE_FLOAT;
  samples[0].timestamp_ms = 0U;
  samples[0].status = 0U;
  strncpy(samples[0].unit, "-", SENSOR_UNIT_LEN);
  samples[0].unit[SENSOR_UNIT_LEN - 1U] = '\0';
  samples[0].value.f32 = 0.0f;

  if (out_count != NULL) {
    *out_count = 1U;
  }

  Xxx_SetStatus(SENSOR_SEVERITY_NORMAL, ERR_OK, 0U);
  return s_status.severity;
}

static Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status)
{
  return Xxx_CopyStatus(status);
}

const Sensor_Driver_t g_xxx_driver =
{
  .device_id = SENSOR_DEVICE_XXX_1,
  .name = "XXX_1",
  .init = Xxx_Init,
  .self_check = Xxx_SelfCheck,
  .read = Xxx_Read,
  .get_status = Xxx_GetStatus
};
```
