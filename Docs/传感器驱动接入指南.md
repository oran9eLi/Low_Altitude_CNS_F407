# 传感器驱动接入指南

本文面向具体传感器驱动开发人员，说明新增一个传感器驱动时需要实现的函数接口、驱动句柄结构体、采样数据填写方式，以及接入传感器注册表的方法。

驱动开发人员只需要遵守本文接口约定，不需要直接修改系统任务、日志、显示、告警或对外通信协议。

## 1. 分层边界

传感器驱动位于 `Sensor_Driver` 层：

```text
App
  -> Sensor_RegistryReadAll()
      -> Sensor_Driver_t.read()
          -> 具体传感器驱动
              -> Bsp UART/CAN/I2C/SPI 等底层接口
```

各层职责：

- `Bsp`：只负责底层收发和板级接口，不解析传感器业务含义。
- `Sensor_Driver`：负责具体传感器初始化、自检、协议解析、数据转换和统一采样输出。
- `App`：负责任务调度、系统状态、告警、日志、显示和对外协议。

驱动中不要直接调用显示、日志、协议发送或 `App_StatusSet()`。

## 2. 文件放置规范

新增传感器驱动时，建议按以下方式命名：

```text
Sensor_Driver/Inc/xxx_driver.h
Sensor_Driver/Src/xxx_driver.c
```

示例：

```text
Sensor_Driver/Inc/gnss_driver.h
Sensor_Driver/Src/gnss_driver.c

Sensor_Driver/Inc/bme280_driver.h
Sensor_Driver/Src/bme280_driver.c
```

驱动文件加入后，还需要把对应 `.c` 文件加入 Keil 工程的 `Sensor_Driver` 分组。

## 3. 必须包含的头文件

驱动头文件至少包含：

```c
#ifndef XXX_DRIVER_H
#define XXX_DRIVER_H

#include "sensor_driver.h"
#include "sensor_device_id.h"

extern const Sensor_Driver_t g_xxx_driver;

#endif
```

驱动实现文件通常包含：

```c
#include "xxx_driver.h"

#include <string.h>
#include "bsp_uart.h"  /* 根据实际总线选择 bsp_uart/bsp_i2c/bsp_can/bsp_spi */
```

## 4. 必须实现的函数签名

每个传感器驱动必须实现以下 4 类函数。

### 4.1 初始化函数

```c
Sensor_Severity_t Xxx_Init(Sensor_Status_t *status);
```

用途：

- 初始化传感器驱动内部状态。
- 配置必要的软件缓冲区。
- 调用 BSP 层准备底层通信接口。

返回值：

- `SENSOR_SEVERITY_NORMAL`：初始化成功。
- `SENSOR_SEVERITY_IMPORTANT`：底层接口未就绪或初始化失败，影响实验或数据可信度。
- `SENSOR_SEVERITY_CRITICAL`：设备异常会影响安全，需要停机或人工处理。

`status` 可以为 `NULL`。如果不为 `NULL`，驱动应填写 `severity`、`reason` 和 `driver_error`。`reason` 用于表达具体故障事实，例如 `SENSOR_FAULT_NOT_READY` 或 `SENSOR_FAULT_INIT_FAILED`。

注意：

- 不要在这里创建 FreeRTOS 任务。
- 不要在这里直接写日志。
- 不要在这里刷新显示。

### 4.2 自检函数

```c
Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status);
```

用途：

- 检查设备是否在线。
- 检查最近数据是否有效。
- 检查通信是否超时。
- 检查设备状态寄存器或协议状态位。

参数：

- `status`：输出驱动状态事实，可以为 `NULL`。其中 `driver_error` 用于定位设备或总线问题，不等同于应用层标准错误码 `App_ErrorCode_t`。

示例：

```c
Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status)
{
  if (status != NULL)
  {
    status->severity = SENSOR_SEVERITY_NORMAL;
    status->reason = SENSOR_FAULT_NONE;
    status->driver_error = 0U;
  }

  if (device_not_ready)
  {
    if (status != NULL)
    {
      status->severity = SENSOR_SEVERITY_IMPORTANT;
      status->reason = SENSOR_FAULT_NOT_READY;
      status->driver_error = 1U;
    }
    return SENSOR_SEVERITY_IMPORTANT;
  }

  return SENSOR_SEVERITY_NORMAL;
}
```

### 4.3 读取函数

```c
Sensor_Severity_t Xxx_Read(Sensor_Sample_t *samples,
                              uint16_t max_count,
                              uint16_t *out_count,
                              Sensor_Status_t *status);
```

用途：

- 从传感器读取或解析最新数据。
- 将数据转换为统一的 `Sensor_Sample_t`。
- 一个驱动可以一次输出多条采样记录。

参数：

- `samples`：调用方提供的采样缓冲区。
- `max_count`：缓冲区最多可写入的采样数量。
- `out_count`：实际写入的采样数量，可以为 `NULL`。
- `status`：读取过程中的状态事实，可以为 `NULL`。

基本要求：

- 如果 `out_count != NULL`，函数开始时应先写 `*out_count = 0U`。
- 如果 `samples == NULL` 或 `max_count == 0U`，返回 `SENSOR_SEVERITY_IMPORTANT`，并把 `reason` 设置为 `SENSOR_FAULT_NOT_READY` 或 `SENSOR_FAULT_NO_DATA`。
- 不允许写超过 `max_count` 的采样数量。
- 没有新数据时按设备语义返回 `SENSOR_SEVERITY_GENERAL` 或 `SENSOR_SEVERITY_IMPORTANT`，并用 `SENSOR_FAULT_NO_DATA` 表达原因。

### 4.4 状态查询函数

```c
Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status);
```

用途：

- 返回驱动当前状态。
- 不触发新的总线读取。
- 用于上层快速查询设备健康状态。

如果当前驱动没有额外状态，可以让它复用自检逻辑：

```c
Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status)
{
  return Xxx_SelfCheck(status);
}
```

## 5. 驱动句柄结构体

每个驱动必须提供一个 `Sensor_Driver_t` 句柄。

定义位置：

```text
Sensor_Driver/Inc/sensor_driver.h
```

结构体：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Type_t sensor_type;
  const char *name;
  Sensor_InitFn init;
  Sensor_SelfCheckFn self_check;
  Sensor_ReadFn read;
  Sensor_GetStatusFn get_status;
} Sensor_Driver_t;
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `device_id` | 设备编号，由项目统一分配，不允许驱动作者随意填裸数字 |
| `sensor_type` | 传感器类型，例如 `SENSOR_TYPE_GNSS`、`SENSOR_TYPE_WEATHER` |
| `name` | 设备名称，用于日志、调试和显示 |
| `init` | 初始化函数 |
| `self_check` | 自检函数 |
| `read` | 采样读取函数 |
| `get_status` | 状态查询函数 |

示例：

```c
const Sensor_Driver_t g_gnss_driver =
{
  .device_id = SENSOR_DEVICE_GNSS_1,
  .sensor_type = SENSOR_TYPE_GNSS,
  .name = "gnss",
  .init = Gnss_Init,
  .self_check = Gnss_SelfCheck,
  .read = Gnss_Read,
  .get_status = Gnss_GetStatus
};
```

## 6. 采样数据填写规范

统一采样结构定义在：

```text
Sensor_Driver/Inc/sensor_data.h
```

结构体：

```c
typedef struct
{
  uint16_t device_id;
  Sensor_Type_t sensor_type;
  uint16_t channel;
  Sensor_ValueType_t value_type;
  uint32_t timestamp_ms;
  uint32_t status;
  char unit[SENSOR_UNIT_LEN];
  Sensor_Value_t value;
} Sensor_Sample_t;
```

字段说明：

| 字段 | 说明 |
| --- | --- |
| `device_id` | 设备编号，必须与驱动句柄一致 |
| `sensor_type` | 传感器类型，必须与驱动句柄一致 |
| `channel` | 通道号，例如温度 0、湿度 1、电压通道 0 |
| `value_type` | 当前采样值类型 |
| `timestamp_ms` | 采样时间戳，单位 ms |
| `status` | 采样状态，0 表示正常 |
| `unit` | 单位字符串，最大长度 `SENSOR_UNIT_LEN` |
| `value` | 采样值联合体 |

### 6.1 浮点值示例

```c
samples[0].device_id = SENSOR_DEVICE_BME280_1;
samples[0].sensor_type = SENSOR_TYPE_WEATHER;
samples[0].channel = 0U;
samples[0].value_type = SENSOR_VALUE_FLOAT;
samples[0].timestamp_ms = timestamp_ms;
samples[0].status = 0U;
strncpy(samples[0].unit, "degC", SENSOR_UNIT_LEN);
samples[0].value.f32 = temperature_deg_c;
```

### 6.2 布尔状态示例

```c
samples[0].device_id = SENSOR_DEVICE_MPU6050_1;
samples[0].sensor_type = SENSOR_TYPE_IMU;
samples[0].channel = 1U;
samples[0].value_type = SENSOR_VALUE_BOOL;
samples[0].timestamp_ms = timestamp_ms;
samples[0].status = 0U;
strncpy(samples[0].unit, "-", SENSOR_UNIT_LEN);
samples[0].value.boolean = imu_calibrated;
```

### 6.3 经纬高示例

```c
samples[0].device_id = SENSOR_DEVICE_GNSS_1;
samples[0].sensor_type = SENSOR_TYPE_GNSS;
samples[0].channel = 0U;
samples[0].value_type = SENSOR_VALUE_GEO;
samples[0].timestamp_ms = timestamp_ms;
samples[0].status = fix_status;
strncpy(samples[0].unit, "geo", SENSOR_UNIT_LEN);
samples[0].value.geo.latitude_deg_e7 = latitude_deg_e7;
samples[0].value.geo.longitude_deg_e7 = longitude_deg_e7;
samples[0].value.geo.altitude_mm = altitude_mm;
```

## 7. 返回值约定

```text
SENSOR_SEVERITY_NORMAL
  正常。

SENSOR_SEVERITY_GENERAL
  可继续运行，但数据质量下降或存在轻微异常。

SENSOR_SEVERITY_IMPORTANT
  影响实验或数据可信度，需要提示或进入降级策略。

SENSOR_SEVERITY_CRITICAL
  影响安全或主系统运行，需要停机或人工处理。
```

严重程度只表达影响级别，不表达具体故障原因。具体原因通过 `Sensor_Status_t.reason` 输出：

```text
SENSOR_FAULT_NONE
  无异常。

SENSOR_FAULT_NOT_READY
  设备或数据尚未就绪。

SENSOR_FAULT_TIMEOUT
  通信或采样超时。

SENSOR_FAULT_BUS
  I2C、UART、SPI 等总线异常。

SENSOR_FAULT_OFFLINE
  设备离线或长时间无响应。

SENSOR_FAULT_NO_DATA
  当前无有效数据。

SENSOR_FAULT_DATA_INVALID
  数据格式、校验或数值无效。

SENSOR_FAULT_NO_FIX
  GNSS 无有效定位。

SENSOR_FAULT_INIT_FAILED
  初始化失败。
```

驱动返回值应尽量稳定，不要因为单次无新数据就频繁返回 `SENSOR_SEVERITY_CRITICAL`。App 层会根据 `sensor_type + reason + driver_error` 统一映射标准错误码和告警项。

## 8. 设备 ID 分配

`device_id` 由项目框架或系统集成侧统一分配。具体传感器驱动只引用已分配的 ID，不自行新增裸数字。

当前定义位置：

```text
Sensor_Driver/Inc/sensor_device_id.h
```

当前已预留：

```c
typedef enum {
  SENSOR_DEVICE_INVALID = 0,
  SENSOR_DEVICE_GNSS_1 = 1,
  SENSOR_DEVICE_MPU6050_1 = 2,
  SENSOR_DEVICE_BME280_1 = 3
} Sensor_DeviceId_t;
```

后续新增 Remote ID、电源、继电器 IO 等设备时，应先扩展该表，再让驱动引用新的枚举值。

## 9. 接入传感器注册表

驱动实现完成后，需要在：

```text
Sensor_Driver/Src/sensor_registry.c
```

包含驱动头文件：

```c
#include "gnss_driver.h"
```

然后把驱动句柄挂入 `s_sensor_drivers[]`：

```c
static const Sensor_Driver_t *const s_sensor_drivers[] =
{
  &g_gnss_driver,
};
```

多个驱动示例：

```c
static const Sensor_Driver_t *const s_sensor_drivers[] =
{
  &g_gnss_driver,
  &g_bme280_driver,
  &g_power_driver,
};
```

接入后，系统会通过注册表统一调用：

```text
Sensor_RegistryInitAll()
Sensor_RegistrySelfCheckAll()
Sensor_RegistrySelfCheckAbnormal()
Sensor_RegistryReadAll()
```

## 10. 异常明细自检接口

系统注册表使用 `Sensor_RegistrySelfCheckAll()` 判断 sensor 模块整体严重程度，但显示、日志、告警和对外协议通常还需要知道具体哪个传感器异常。

因此传感器注册表直接输出 `Sensor_Status_t` 异常明细：

字段说明：

| 字段 | 说明 |
| --- | --- |
| `device_id` | 异常传感器设备编号 |
| `sensor_type` | 异常传感器类型 |
| `severity` | 异常严重程度，例如 `SENSOR_SEVERITY_GENERAL`、`SENSOR_SEVERITY_IMPORTANT` |
| `reason` | 传感器域内故障原因，例如 `SENSOR_FAULT_NO_FIX`、`SENSOR_FAULT_TIMEOUT` |
| `driver_error` | 具体驱动返回的内部细节码，不直接作为 HMI/日志/上报的标准错误码 |

异常明细接口：

```c
Sensor_Severity_t Sensor_RegistrySelfCheckAbnormal(Sensor_Status_t *status_list,
                                                   uint16_t max_count,
                                                   uint16_t *out_count);
```

使用方式：

```c
Sensor_Status_t abnormal_list[8];
uint16_t abnormal_count = 0U;

Sensor_Severity_t result = Sensor_RegistrySelfCheckAbnormal(abnormal_list,
                                                            8U,
                                                            &abnormal_count);
```

约定：

- 只输出异常传感器，正常传感器不会写入 `status_list`。
- `out_count` 返回本次发现的异常总数。
- 如果 `out_count > max_count`，表示调用方缓冲区不足，只保存了前 `max_count` 条异常。
- 返回值仍然是所有异常中的最严重结果。

驱动开发人员不需要直接调用这个接口，但需要保证自己的 `self_check()` 正确返回 `severity`，并填写 `Sensor_Status_t.reason` 和 `driver_error`。最终对外错误码由应用层映射为 `App_ErrorCode_t`。

## 11. 禁止事项

具体传感器驱动中不要做以下事情：

- 不要直接创建 FreeRTOS 任务。
- 不要直接写日志文件。
- 不要直接刷新 DWIN 显示。
- 不要直接拼接主控到网关的协议帧。
- 不要直接调用 `App_StatusSet()`、`App_AlarmRaise()` 或 `App_AlarmPost()`。
- 不要在驱动接口中暴露 HAL 句柄。
- 不要在驱动里使用未统一分配的 `device_id` 裸数字。

如果驱动发现异常，应通过 `Sensor_Severity_t` 和 `Sensor_Status_t` 上报，由注册表和应用层统一映射为标准错误码、告警、日志和显示。

## 12. 最小驱动模板

```c
#include "xxx_driver.h"

#include <string.h>
#include "sensor_device_id.h"

static Sensor_FaultReason_t s_fault_reason;
static uint32_t s_driver_error;

static void Xxx_FillStatus(Sensor_Status_t *status, Sensor_Severity_t severity)
{
  if (status == NULL)
  {
    return;
  }

  status->severity = severity;
  status->reason = s_fault_reason;
  status->driver_error = s_driver_error;
}

Sensor_Severity_t Xxx_Init(Sensor_Status_t *status)
{
  s_fault_reason = SENSOR_FAULT_NONE;
  s_driver_error = 0U;
  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_SelfCheck(Sensor_Status_t *status)
{
  if (s_driver_error != 0U)
  {
    Xxx_FillStatus(status, SENSOR_SEVERITY_GENERAL);
    return SENSOR_SEVERITY_GENERAL;
  }

  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_Read(Sensor_Sample_t *samples,
                           uint16_t max_count,
                           uint16_t *out_count,
                           Sensor_Status_t *status)
{
  if (out_count != NULL)
  {
    *out_count = 0U;
  }

  if ((samples == NULL) || (max_count == 0U))
  {
    s_fault_reason = SENSOR_FAULT_NOT_READY;
    s_driver_error = 1U;
    Xxx_FillStatus(status, SENSOR_SEVERITY_IMPORTANT);
    return SENSOR_SEVERITY_IMPORTANT;
  }

  samples[0].device_id = SENSOR_DEVICE_XXX_1;
  samples[0].sensor_type = SENSOR_TYPE_UNKNOWN;
  samples[0].channel = 0U;
  samples[0].value_type = SENSOR_VALUE_FLOAT;
  samples[0].timestamp_ms = 0U;
  samples[0].status = 0U;
  strncpy(samples[0].unit, "-", SENSOR_UNIT_LEN);
  samples[0].value.f32 = 0.0f;

  if (out_count != NULL)
  {
    *out_count = 1U;
  }

  s_fault_reason = SENSOR_FAULT_NONE;
  s_driver_error = 0U;
  Xxx_FillStatus(status, SENSOR_SEVERITY_NORMAL);
  return SENSOR_SEVERITY_NORMAL;
}

Sensor_Severity_t Xxx_GetStatus(Sensor_Status_t *status)
{
  return Xxx_SelfCheck(status);
}

const Sensor_Driver_t g_xxx_driver =
{
  .device_id = SENSOR_DEVICE_XXX_1,
  .sensor_type = SENSOR_TYPE_UNKNOWN,
  .name = "xxx",
  .init = Xxx_Init,
  .self_check = Xxx_SelfCheck,
  .read = Xxx_Read,
  .get_status = Xxx_GetStatus
};
```

模板中的 `SENSOR_DEVICE_XXX_1` 需要由统一设备编号表提供。
